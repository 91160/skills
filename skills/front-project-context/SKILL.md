# Front Project Context Skill

帮助用户对项目进行规范提取，生成核心文件：
- `FRONT_PROJECT_CONTEXT.md`：项目知识地图，供模型每次生成代码前查阅

## 使用前提

- 需要文件系统访问权限（Claude Code 直接使用；Claude.ai 需用户上传项目文件或在容器中准备好项目）
- 支持前端项目（Vue / React / Next.js / Nuxt.js 等）及其他类型项目

---

## 核心执行原则

**能从文件读到的信息，绝对不问用户。**

只有以下三种情况才允许询问用户：
1. 文件中存在真实歧义需要人工判断（如同时检测到 Vue2 和 Vue3 的依赖）
2. 需要用户指定"基准文件"（统计结果无法判断哪些是团队认可的标准实现）
3. 自动探测全部完成后，做一次最终核对确认（一次性，放在扫描开始前）

---

## 执行流程

### 第一步：自动探测项目基本情况

**不询问用户，直接读取以下文件：**

```
1. package.json          → 框架、UI库、工具链、版本号
2. 所有 README.md        → 项目说明、模块约定、特殊规范、目录说明
3. vite.config.* / next.config.* / vue.config.*  → 构建配置、路径别名
4. .eslintrc* / .prettierrc*  → 代码风格约定
5. tailwind.config.* / uno.config.*  → 样式配置
6. docs/ 目录            → 已有规范文档
7. .cursorrules / CLAUDE.md  → 已有 AI 规范约定
8. tsconfig.json         → TS 配置、路径映射
```

**从以上文件中自动提取：**
- 技术栈（框架 / UI库 / 状态管理 / 请求封装 / 样式方案）
- 已有规范文档位置
- 路径别名映射（如 `@` → `src/`）
- 已有的编码约定（lint 规则等）

**探测完成后，向用户展示探测结果摘要，并只问一个问题：**

> "以上是我从项目文件中自动识别的基本情况，有需要补充或纠正的吗？若没有，我将开始全量扫描。"

收到用户确认后，进入第二步。

---

### 第二步：分轮扫描（每轮只做一件事）

**关键原则：先统计，再归纳。不允许"看几个文件就下结论"。**

每个模块分两轮执行：

**采样轮**：扫描目标目录下所有文件，列出原始代码片段，不做任何归纳评价。

**归纳轮**：基于采样结果，统计各写法出现频率，按以下标准分类：
- 主流写法（> 60%）→ 定为规范
- 少数写法（< 20%）→ 标注为「遗留代码，待重构」
- 混用写法 → 标注为「需团队决策」

**验证轮**：从未参与采样的文件中随机抽查 5 个，验证归纳结果准确性，输出偏差报告。若该模块剩余未采样文件不足 5 个，则对剩余文件全量验证；若采样已覆盖全部文件，则跳过验证轮。

> 详细的各模块扫描指令见 `references/scan-modules.md`

---

### 第三步：生成 FRONT_PROJECT_CONTEXT.md

在项目根目录生成此文件，结构如下：

```markdown
# FRONT_PROJECT_CONTEXT.md
> 此文件由 Claude 自动生成，供 AI 在生成代码时查阅，请勿手动大幅修改。
> 最后更新：{日期}

## 1. 技术栈总览
- 框架：
- UI 组件库：
- 状态管理：
- 请求封装：
- 样式方案：

## 2. 常量速查表
| 常量名 | 值 | 路径 | 用途 |
|-------|-----|------|------|

## 3. 请求方法
- 封装位置：
- 可用方法签名：
- 拦截器说明：
- 真实调用示例：

## 4. 公共方法速查表
| 函数名 | 签名 | 路径 | 用途 |
|-------|------|------|------|

## 5. 全局组件清单
| 组件名 | Props 摘要 | 使用场景 |
|-------|-----------|---------|

## 6. 路由结构摘要
（一级 + 二级路由，标注权限路由）

## 7. 环境变量清单
| 变量名 | 用途 |（不输出真实值）

## 8. 禁止事项
基于已有封装，明确列出禁止行为，例如：
- ❌ 禁止直接使用 axios，统一用 src/utils/request.ts
- ❌ 禁止硬编码接口地址，统一走 constants/api.ts
- ❌ 禁止直接使用 dayjs，统一用 src/utils/date.ts 中的 formatDate()
```

---

### 第四步：输出规范总结

生成完成后，向用户输出：

1. 文件的路径确认
2. 发现的主要规范条目摘要（前 10 条）
3. 发现的问题清单（遗留代码、混用写法），附文件路径
4. 建议优先统一的 Top 3 问题

---

### 增量更新机制

当用户再次运行此技能时（如项目代码有更新），按以下策略执行：

1. **检测已有文件**：检查项目根目录是否已存在 `FRONT_PROJECT_CONTEXT.md`
2. **差异扫描**：如 `FRONT_PROJECT_CONTEXT.md` 已存在，则对比当前扫描结果与已有内容，仅更新有变化的章节（新增、删除、修改的常量 / 方法 / 组件等）
3. **变更日志**：在 `FRONT_PROJECT_CONTEXT.md` 头部的「最后更新」字段旁追加变更摘要，例如：
   ```
   > 最后更新：2025-06-15（本次更新：新增 3 个公共方法，移除 2 个废弃常量，更新请求封装签名）
   ```
4. **向用户展示 diff 摘要**：列出本次更新了哪些章节、新增/删除/修改了哪些条目，让用户确认

## 重要执行原则

| 原则 | 说明 |
|------|------|
| **自动探测优先** | 能从文件读到的信息绝对不问用户，package.json / README.md / config 文件是第一信源 |
| **README 必读** | 项目根目录及各子目录的 README.md 必须在扫描前全部读取，其中的约定优先级高于代码统计结果 |
| 用户指定基准 | 用户指定的"标准文件"优先级高于统计结果 |
| 分离提取与归纳 | 不允许边读边总结，必须先采样再统计 |
| 区分规范与遗留 | 发现问题代码时明确标注，不将其纳入规范 |
| 不输出敏感信息 | 环境变量只列变量名和用途，不输出真实值 |
| 歧义才询问 | 只有自动探测产生真实歧义时才打断用户，不因"不确定"就开口问 |
