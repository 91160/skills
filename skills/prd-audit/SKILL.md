# PRD 需求审计 Skill

对 PRD 文档执行结构化审计，覆盖完整性、一致性、可行性、边界清晰度、优先级合理性、安全与合规六大维度，输出标准化审计报告。

## 适用范围

先判断项目类型，选择对应的审计模式：

| 项目类型 | 审计方式 |
|---------|---------|
| 新项目 | **完整审计** — 执行全部 6 个维度、所有检查项 |
| 旧项目新需求 | **轻量验证** — 仅执行完整性 + 一致性 + 可行性 + 边界清晰度 |
| Bug 修复 / 技术优化 | 不适用，告知用户此 skill 不覆盖此类需求 |

如果用户没有说明项目类型，默认按「新项目完整审计」执行。如果不确定，可以先问用户。

## 工作流程

### Step 1: 获取 PRD 内容

- 如果用户上传了文件（.md / .pdf / .docx / .txt），先读取文件内容
  - 对于 PDF 文件，参考 `/mnt/skills/public/pdf-reading/SKILL.md` 中的方法提取文本
  - 对于 DOCX 文件，参考 `/mnt/skills/public/file-reading/SKILL.md` 中的方法读取
- 如果用户直接在对话中粘贴了 PRD 内容，直接使用
- 如果用户没有提供 PRD，提示用户上传或粘贴

### Step 2: 识别项目类型

根据以下线索判断：
- 用户明确说明（"这是新项目" / "这是已有系统的新需求"）
- PRD 中是否提到现有系统、迁移、兼容等关键词 → 可能是旧项目新需求
- 无法判断时询问用户，或默认按新项目处理

### Step 3: 逐维度审计

打开参考文件 `references/audit-rules.md` 获取完整审计规则，然后按维度逐一检查。

审计规则文件路径：读取与本 SKILL.md 同目录下的 `references/audit-rules.md`。

对每个检查项，给出以下判定之一：
- **✅ 通过** — PRD 在此项上描述充分
- **⚠️ 发现问题** — 存在缺陷，需标注严重程度（P0/P1/P2）并给出修复方案
- **❓ 无法判断** — PRD 未涉及此领域，无法评估（标注原因）

审计时注意：
- P0 问题必须给出明确修复方案，不能含糊带过
- P1 问题应给出建议方案或 A/B 选项供用户选择
- 同一个问题可能涉及多个维度，选择最匹配的维度归类，避免重复列出
- 如果 PRD 篇幅较短或信息有限，应如实反映「信息不足以判断」，而不是强行通过

### Step 4: 汇总并输出报告

根据审计结果，按以下逻辑确定结论：

| 结论 | 条件 |
|------|------|
| **PASS** | P0 = 0 且 P1 ≤ 5 |
| **WARNING** | P0 = 0 且 P1 > 5 |
| **REJECT** | P0 ≥ 1 |

输出格式严格遵循下方模板。

## 审计报告输出模板

```
【PRD 需求审计报告】

项目类型：{新项目 | 旧项目新需求}
审计模式：{完整审计 | 轻量验证}
审计结论：{PASS | WARNING | REJECT}
缺陷统计：P0={N} | P1={N} | P2={N}

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[通过项]
- {检查项编号} | PRD {章节}：{一句话说明} ✅

[发现项]
- AUDIT-{序号} | 维度 {A~F} | {检查项编号} | {P0|P1|P2} | PRD {章节}
  问题：{一句话描述}
  修复方案：{技术确定性修复 或 A/B 方案供用户选择}

[风险项]
- RISK-{序号} | {风险描述} | {影响范围} | {P0|P1|P2}
  应对措施：{建议方案}

[技术待决策项]
- TECH-{序号} | {待决策项} | 影响范围: {哪些模块依赖此决策}

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

下一步建议：
- {根据审计结论给出的后续行动建议}
```

### Step 5: 后续交互

报告输出后，提示用户：
- 如果结论是 REJECT：指出哪些 P0 问题必须优先修复，可以帮助用户一起修改 PRD
- 如果结论是 WARNING：列出 P1 问题并询问用户处理意向
- 如果结论是 PASS：恭喜用户，建议进入下一阶段（Spec 编写 / 技术方案设计等）

## 严重程度参考

| 级别 | 含义 | 典型场景 |
|------|------|---------|
| P0 — 阻塞 | 逻辑矛盾、核心缺失、安全漏洞 | 核心流程未定义、状态机矛盾、敏感数据无加密方案 |
| P1 — 歧义 | 描述不清、边界未定义、数值缺失 | 接口参数类型未说明、并发量无指标、角色权限未定义 |
| P2 — 细节 | 命名不规范、描述可优化 | 术语不统一但不影响理解、缺少示例但逻辑清晰 |

## 注意事项

- 保持客观中立，基于 PRD 文本本身审计，不做过度推测
- 审计报告的语气应专业但友好，像一个经验丰富的技术 Lead 在做 code review
- 如果 PRD 整体质量很高，也要如实反映，不要为了"找问题"而强行挑刺
- 对于不同行业/领域的 PRD，审计标准一致，但举例和建议应贴合领域上下文
