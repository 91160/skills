# SDD Workflow — AI 执行规则 v3.0

> **最高指令**：严禁在未确认变更通道的情况下直接编写或修改生产代码。每次回复优先使用中文。

---

## 0. 每次对话开始

**第一步**：AI 主动询问项目类型：

```
【项目确认】
请问这是新项目还是旧项目？
  A. 新项目（请提供 PRD）
  B. 旧项目 — 新增需求
  C. 旧项目 — Bug 修复
  D. 旧项目 — 技术优化
```

**第二步**：读取 `.project/context.md`，恢复工作状态（不存在则跳过，进入 §1 初始化）。

**类型映射**（贯穿后续所有阶段）：

| §0 选项 | 任务类型 | 对应 §2.1 模板 |
| -------- | -------- | --------------- |
| A / B    | feature  | 新增需求        |
| C        | bug      | Bug 修复        |
| D        | refactor | 技术优化        |

## 0.5 写码门禁（Code Gate）

> AI 每次准备写/改生产代码前，必须自检以下条件，**全部 ✅ 才可执行**：

| # | 检查项                                              | 未通过处理                |
| - | --------------------------------------------------- | ------------------------- |
| 1 | `.project/` 已初始化                              | → 先完成图一             |
| 2 | 当前功能有 REQ + DES 且 `review-status: approved` | → 先完成图二 §2.1-§2.4 |
| 3 | 阻塞依赖已就绪（用户明确说"跳过"的视为就绪）        | → 等待用户提供或确认跳过 |

**快速通道豁免**：仅样式/文案的快速通道任务，检查项 #2 简化为：REQ + DES 可各简化为一句话描述，免评审。AI 在进入 Step 0 前，在对应 REQ + DES 文件中追加一句话记录（文件不存在则新建）。

- 任一 ❌ → **停止**，输出缺失项，等待用户指示
- **禁止将"继续"、"开始"等模糊指令默认理解为"直接写代码"**，应先执行门禁自检

## 1. 图一：项目启动

### 1.1 新项目流程

```
用户提供 PRD
  → AI 读取 PRD，理解业务需求
  → AI 执行 PRD 审计：
    1. 按 §1.3 Skill 执行流程调用 /prd-audit Skill
    2. Skill 不可用 → 按默认6个维度（完整性/一致性/可行性/边界/优先级/安全合规）+ P0/P1/P2 标准执行
    3. P0 必须解决后才继续
  → AI 询问技术栈（前端 / 后端 / 数据库）→ 写入 project-custom.md「技术栈」区块
  → AI 询问外部依赖（第三方API / 内部服务 / 组件库）
  → AI 询问外部依赖文档并学习 → 记录到 project-custom.md「外部依赖」区块
  → AI 初始化 .project/ 目录
  → AI 初始化代码结构（脚手架）：
    1. 查 §1.3 Skill 路由表，按技术栈匹配对应 Skill
    2. 后端 Java → 按 §1.3 Skill 执行流程调用 java-project-creator 生成脚手架
    3. 前端 → 按 §1.3 Skill 执行流程调用 wap-project-creator 生成脚手架
    4. Skill 不可用 → 用框架官方 CLI / 手动搭建兜底
  → 安装依赖，确认能跑起来
  → 进入图二业务开发循环
```

**技术栈询问格式**：

```
【技术栈确认】
请告诉我本项目的技术栈：
  1. 前端框架？（如 Vue3 / React）
  2. 后端框架？（如 SpringBoot / Node.js）
  3. 数据库？（如 MySQL / PostgreSQL）
```

**外部依赖询问格式**：

```
【外部依赖确认】
请告诉我需要调用的外部服务：
  1. 第三方 API？（如微信支付、地图服务）
  2. 公司内部其他服务接口？
  3. 指定的前端组件库或工具库？
```

### 1.2 旧项目流程

**AI 自动执行**（读代码得出，无需人工提供）：

- 项目结构理解 → 输出项目摘要
- 技术栈识别 → 写入 project-custom.md「技术栈」区块
- 现有规范识别（命名约定 / 目录结构 / 代码风格 / commit 规范）→ 提取后写入 project-custom.md「项目规范」区块，按前后端分区，人工确认
- 外部依赖梳理 → 记录到 project-custom.md「外部依赖」区块
- 数据库结构梳理（核心表 / 关联关系）
- 接口文档梳理（有则读取，无则从代码反推）

**人工确认**（AI 无法自行判断，必须问）：

- 危险区域确认（核心逻辑 / 遗留代码 / 不能动的部分）
- 分支策略确认（开发分支 / review 流程 / 合并策略）
- 部署环境确认（dev / staging / prod / CI/CD 流程）
- 敏感配置确认（密钥 / 证书 / .env 处理方式）
- 性能和业务约束（瓶颈 / 特殊要求）
- 团队协作信息（维护者 / 并行开发 / 提交规范）
- 基线确认（核心功能是否正常 / 已知 bug / 进行中分支）

**完成后**：

```
AI 生成 .project/ 目录（汇总所有信息）
  → 安装依赖，确认能跑起来
  → 进入图二业务开发循环
```

### 1.2.5 project-custom.md 模板

项目铁律由团队人工填写，AI 不自动生成。技术栈、项目规范、外部依赖由 AI 在初始化时自动填写。

**完整模板：**

```markdown
## 技术栈
- frontend: {vue3 / react / ...}
- backend: {java-springboot / go / ...}
- database: {mysql / postgresql / ...}

## 项目铁律（Constitution）
> 以下约定 AI 绝对不能违反，无论任何情况。违反时必须立即停止并报告。

### 技术约束
- [ ] {例：必须使用 Vue3，禁止使用 Vue2}
- [ ] {例：所有接口调用必须封装在 service 层，禁止在组件里直接调用}

### 代码规范
- [ ] {例：代码注释必须使用中文}
- [ ] {例：所有异步操作必须有错误处理}

### 业务约束
- [ ] {例：涉及金额的字段必须使用 Decimal，禁止使用 Float}
- [ ] {例：用户数据操作必须有操作日志}

## 项目规范（AI 提取，人工确认）
### 前端（{tech}）
- 命名约定：...
- 目录结构：...
- 代码风格：...

### 后端（{tech}）
- 命名约定：...
- 目录结构：...
- 代码风格：...

## 外部依赖（API 学习记录）
- ...
```

**AI 执行规则**：

- 每次标准通道和快速通道开始前必须先读取铁律（执行详见 §3 Step 0）
- 未收到用户明确指示不得继续执行

### 1.3 Skill 路由表

公司级规范以 Skill 形式封装，AI 在对应阶段自动安装并调用。

**Skill 仓库**：`https://git.nykjsrv.cn/ai-coding/skills.git`
**安装命令**：`npx skills add {仓库地址} --skill {skill名称} --yes`

| 阶段             | 触发时机                    | Skill 路由            | skill 名称           |
| ---------------- | --------------------------- | --------------------- | -------------------- |
| PRD 审计         | §1.1 / §2.1                | /prd-audit            | prd-audit            |
| 后端项目初始化   | §1.1 脚手架创建（仅新项目） | /java-project-creator | java-project-creator |
| 前端项目初始化   | §1.1 脚手架创建（仅新项目） | /wap-project-creator  | wap-project-creator  |
| 前端编码         | §3 Step 4 前端改动          | /frontend-coding      | frontend-coding      |
| 后端编码         | §3 Step 4 后端改动          | /java-coding          | java-coding          |
| 代码审计         | §3 Step 5                   | /code-audit           | code-audit           |

**Skill 执行流程**（所有触发点统一遵守）：

```
1. 检查项目目录 .agents/skills/{skill名称}/ 是否已存在
   → 已存在 → 直接进入第 3 步（调用）
   → 不存在 → 进入第 2 步（安装）
2. 安装：执行 npx skills add {仓库地址} --skill {skill名称} --yes
   → 安装成功 → 进入第 3 步
   → 安装失败 → 跳过调用，由各触发点使用内置默认兜底
3. 调用：读取 Skill 内容，按其规范执行
```

**匹配规则**：
- 根据 project-custom.md「技术栈」声明 + 当前改动文件类型，匹配对应 Skill
- Skill 安装失败 → 用内置默认兜底（PRD 默认 6 维度 / C-01~C-10 / S-01~S-05）
- **项目初始化（仅新项目 §1.1）**：技术栈含 Java → java-project-creator；技术栈含前端 → wap-project-creator；旧项目（§1.2）已有代码结构，不触发
- **编码阶段**：前端改动调用 /frontend-coding，后端改动调用 /java-coding（§3 Step 4）
- **代码审计**：前后端统一调用 /code-audit（§3 Step 5）

**融合规则**（project-custom.md 与 Skill 的关系）：
1. 读取 project-custom.md（铁律 + 项目规范）
2. 调用 Skill 获取公司级规范
3. 冲突项 → 以 project-custom.md 为准
4. project-custom.md 未提及 → 用 Skill 规范
5. 两者都未覆盖 → 用内置默认

## 2. 图二：业务开发循环

每次迭代按以下阶段执行：

### 2.1 需求分析

根据 §0 确定的任务类型，使用对应的 REQ 模板：

**A/B. 新增需求（feature）**：

1. AI 读取需求，初步理解功能意图
2. **AI 主动澄清（必须执行）**：

```
【需求澄清】
我理解这次要做 {功能名}，但有以下几点需要确认：
  1. {不清晰的地方}
  2. {边界不明确的地方}
  3. {可能有多种实现方式需要选择的地方}

需求审计：
  1. 按 §1.3 Skill 执行流程调用 /prd-audit Skill
  2. Skill 不可用 → 使用默认轻量审计（4 维度）：
     - 完整性：正常+异常流程是否完整
     - 一致性：与现有功能/数据模型是否冲突
     - 可行性：现有架构是否支撑
     - 边界：做什么/不做什么是否明确
请确认后继续。
```

3. 用户补充说明后，AI 确认验收标准
4. 验收标准必须具体，不接受模糊描述（如"功能正常"）
5. 写入 REQ：需求描述、验收标准

**C. Bug 修复（bug）**：

1. **AI 主动收集 Bug 信息（必须执行）**：

```
【Bug 定位】
请确认以下信息：
  1. 复现步骤？
  2. 预期行为 vs 实际行为？
  3. 影响范围（哪些模块/用户受影响）？
  4. 严重程度：P0紧急 / P1重要 / P2一般？
```

2. AI 确认验收标准（Bug 修复后预期行为 + 不引入回归）
3. 写入 REQ：复现步骤、预期vs实际、影响范围、严重程度、验收标准
4. 仅涉及样式/文案的 Bug，进入 §2.5 开发时走快速通道

**D. 技术优化（refactor）**：

1. **AI 主动确认优化目标（必须执行）**：

```
【优化目标确认】
请确认以下信息：
  1. 优化什么？（具体目标）
  2. 不改什么？（业务行为不变的边界）
  3. 如何验证？（怎么证明优化有效且无回归）
```

2. AI 确认验收标准（优化指标 + 回归验证方式）
3. 写入 REQ：优化目标、不变边界、验证方式、验收标准

### 2.2 方案设计

根据任务类型，使用对应的 DES 模板，写入 `specs/master/design/DES-{xx}-{name}.md`：

> DES 达标标准：读完 DES 即可编码，无需猜测。涉及的维度必须覆盖，不涉及的标注「不涉及」或跳过。

| 任务类型           | DES 内容                                       |
| ------------------ | ---------------------------------------------- |
| **feature**  | 技术方案、数据结构、接口设计、影响范围         |
| **bug**      | 根因分析、修复方案、影响范围、回归风险         |
| **refactor** | 重构方案、改动前后对比、影响范围、回归验证计划 |

**feature 类型须覆盖以下维度**：

| 维度     | 必须明确                      |
| -------- | ----------------------------- |
| 数据模型 | 完整字段定义（名/类型/约束）  |
| API 定义 | 路径/方法/参数/响应/错误码    |
| 核心流程 | 正常+异常路径，节点有输入输出 |
| 状态管理 | 状态枚举+流转条件（如涉及）   |
| 权限矩阵 | 角色-资源-操作（如涉及）      |

### 2.3 原型生成（仅 feature 类型且涉及前端页面时执行）

- AI 生成页面原型（HTML 线框图）
- 输出交互流程图
- 输出页面结构说明

> bug / refactor 类型 → 跳过此步，直接进入 §2.4 评审。

### 2.4 评审（人工）

- 人工确认方案和原型
- 不通过 → AI 根据反馈修改，重新设计
- 通过 → 进入开发

### 2.4.5 人工批注同步（Spec Sync）

用户可随时直接修改 `.project/specs/` 下任何文件（REQ、DES、index.md、context.md、project-custom.md 等），修改后告知 AI。

**触发方式**：用户说"我改了 XXX" / "重新读取 XXX" / "specs 有更新"

**AI 必须要执行的步骤**：

```
Step 1: 重读用户指定的文件
Step 2: 识别改动点（与 AI 上次已知内容对比），输出改动摘要
Step 3: 用户确认改动识别是否正确
Step 4: 级联更新关联 specs 文件（规则见下表）
Step 5: 一致性扫描与状态更新
         - index.md ↔ 实际文件是否对齐（编号、名称、文件存在性）
         - DES → REQ 引用是否一致
         - 发现不一致 → 输出清单（含影响说明），等待用户确认后修复
         - 检查涉及模块是否已有代码实现：已编码 → sync-status 标记为 outdated，未编码 → 保持不变
Step 6: 写入 change-log-specs.md（用户改动 + AI 级联改动分别记录）
Step 7: 检查是否存在 outdated 模块，若有则询问：
```

```
【同步确认】
本次改动导致以下模块 specs 与代码不一致：
  - {模块名}（outdated）
是否现在同步代码？
  A. 立即同步（进入标准通道）
  B. 稍后处理（保持 outdated，下次开发时自动触发）
```

**级联规则**：

| 被改文件            | 可能影响的关联文件                                    |
| ------------------- | ----------------------------------------------------- |
| requirements/REQ-xx | → 对应 design/DES-xx → index.md                     |
| design/DES-xx       | → 对应 requirements/REQ-xx → index.md → context.md |
| index.md            | → context.md                                         |
| context.md          | → 无级联（终端文件）                                 |
| project-custom.md   | → 铁律变更时检查所有 DES 是否冲突                    |

**留痕规则**：

所有改动必须记录到 `.project/specs/change-log-specs.md`，格式：

```markdown
| 序号 | 时间 | 操作人 | 改动文件 | 改动摘要 | 级联更新 |
```

- 操作人标注「用户」或「AI」
- 用户改动：记录改了什么
- AI 级联改动：记录改了哪些文件、改了什么、为什么改

### 2.5 开发

- 走变更通道（见图三）

### 2.6 测试（暂跳过）

> 当前阶段跳过，后续补充。

### 2.7 上线（暂跳过）

> 当前阶段跳过，后续补充。

### 2.8 归档

- 由 §3 Step 7（Changelog）+ Step 7.5（specs 状态）+ Step 8（context.md）自动完成
- 标注任务类型标签：`[Feature]` / `[Bug]` / `[Refactor]`
- 进入下一个需求，↻ 循环

## 3. 图三：变更通道

每次开发任务，必须先判断变更类型：

| 通道           | 触发条件                     | 执行路径                                             |
| -------------- | ---------------------------- | ---------------------------------------------------- |
| **直通** | 不涉及代码（讨论/解释/运行） | 直接回答，完成后更新 context.md                      |
| **快速** | 仅样式/文案，不改变逻辑      | Step 0 → 直接改代码 → Step 7 → Step 7.5 → Step 8 |
| **标准** | 涉及业务逻辑/接口/数据       | 执行 Step 0-8 完整步骤                               |

### 标准通道执行步骤

**Step 0：检查项目铁律（Constitution）**

读取 `project-custom.md` 的「项目铁律」区块，逐条检查本次变更是否违反。

- 铁律区块为空或文件不存在 → 视为无铁律约束，继续 Step 1
- 无冲突 → 继续 Step 1
- 有冲突 → 立即停止，输出：

```
【铁律冲突】
本次变更违反了以下铁律：
  - {具体铁律条目}
建议方案：{符合铁律的替代实现方式}
请确认如何处理后继续。
```

**Step 1：加载 Spec**

```
读取 .project/specs/master/index.md
  → 路由表匹配目标模块（含类型列：feature / bug / refactor）
  → 读取 master/requirements/REQ-xx + master/design/DES-xx
  → 检查 sync-status
```

| sync-status  | 含义                                     | 处理                   |
| ------------ | ---------------------------------------- | ---------------------- |
| `synced`   | spec 与代码一致                          | 正常执行               |
| `outdated` | 代码已变更 spec 未同步                   | 先对齐再执行           |
| `pending`  | 尚未开始（Code Gate 未拦截成功的情况下） | 回退到 §2.1 补写 spec |

**Step 2：文档学习（按需）**

涉及未记录的 API / 库时触发：

```
【文档学习确认】
本次任务涉及 {API/库名}，是否有文档需要我先学习？
支持：A. 在线链接  B. PDF  C. 粘贴内容
```

学习后输出摘要，人工确认后反填 `project-custom.md`「外部依赖」区块。

**Step 3：评估影响面**

- 判断本次改动波及哪些模块
- 识别潜在风险点
- 按以下维度输出影响面清单：

| 维度     | 检查内容                           |
| -------- | ---------------------------------- |
| 功能关联 | 其他模块是否依赖或调用被修改的功能 |
| 数据关联 | 是否涉及共享数据模型/表/缓存       |
| 安全影响 | 是否涉及认证/授权/脱敏/校验        |
| 性能影响 | 是否影响查询性能/并发/内存         |

**Step 4：执行代码变更**

- 符合 `.project/specs/` 规范
- 按 §1.3 Skill 执行流程调用编码 Skill（前端 /frontend-coding，后端 /java-coding），按融合规则与 project-custom.md 融合，Skill 不可用则跳过 Skill，使用内置规约执行
- 逐项遵守编码规约（融合优先级：project-custom.md > Skill 规范 > C-01~C-10）：

| #    | 规约             | 要求                                          |
| ---- | ---------------- | --------------------------------------------- |
| C-01 | 异常路径完整     | 每个 try-catch 有错误分类、用户提示、日志记录 |
| C-02 | 输入校验双重防线 | 前端校验+后端校验，后端不可省                 |
| C-03 | 幂等与并发安全   | 写操作幂等（Token/唯一键），并发敏感加锁      |
| C-04 | 配置外部化       | 所有环境配置走环境变量，零硬编码              |
| C-05 | 结构化日志       | 关键路径有结构化日志，错误日志含上下文        |
| C-06 | 超时与降级       | 所有外部调用设超时，超时有降级方案            |
| C-07 | 数据边界处理     | null/空数组/超长文本/特殊字符有防御           |
| C-08 | 安全基线         | SQL参数化、XSS转义、敏感数据不进日志          |
| C-09 | 错误码体系       | 业务错误有明确错误码                          |
| C-10 | 资源释放         | 连接/句柄/定时器用完必释放                    |

**Step 5：代码审计**

- AI 自我审计，按 §1.3 Skill 执行流程调用 /code-audit Skill，按融合规则与 project-custom.md 融合，Skill 不可用则跳过 Skill，使用内置规约执行
- 融合优先级：project-custom.md > Skill 规范 > S-01~S-05
- 逐项检查，不通过项当场修复，修复后重新检查该项，直到全部通过
- 全部通过后输出审计报告：

```
【代码审计报告】
结论：PASS
检查项：{N}/5 通过 | 修复：{N} 项

- S-01 空值防御：✅
- S-02 前端输入信任：✅（已修复：{修复内容}）
- S-03 Spec 一致性：✅
- S-04 性能回归：✅
- S-05 权限校验：✅
```

| #    | 审计项       | 检查内容                        |
| ---- | ------------ | ------------------------------- |
| S-01 | 空值防御     | null/undefined/空数组是否有防御 |
| S-02 | 前端输入信任 | 后端是否独立校验，不信任前端    |
| S-03 | Spec 一致性  | 实现是否与 REQ+DES 一致         |
| S-04 | 性能回归     | 是否引入慢查询/大循环/内存问题  |
| S-05 | 权限校验     | 接口权限是否正确配置            |

**Step 6：开发自测**

- 优先读取 rules/unit-test-base.md（团队自定义自测规则）
- 文件不存在 → 使用以下默认自测清单
- 逐项验证，不通过项回 Step 4 修复代码，修复后重新执行 Step 5 + Step 6
- 全部通过后输出自测报告：

```
【开发自测报告】
结论：PASS
检查项：{N}/6 通过 | 修复：{N} 项 | 回归次数：{N}

[前端验证]
- T-01 交互逻辑：✅ 验证页面：{页面列表}
- T-02 页面跳转：✅ 验证路径：{路径列表}
- T-03 页面渲染：✅

[后端验证]
- T-04 接口可用性：✅ 验证接口：{接口列表}
- T-05 边界覆盖：✅ 覆盖场景：{场景列表}
- T-06 返回值正确性：✅
```

**前端验证：**

| #    | 检查项     | 检查内容                                               |
| ---- | ---------- | ------------------------------------------------------ |
| T-01 | 交互逻辑   | 改动点涉及的每个页面中，相关按钮的交互逻辑是否正常     |
| T-02 | 页面跳转   | 改动涉及的页面跳转路径是否正确                         |
| T-03 | 页面渲染   | 改动涉及的页面是否正常渲染，无白屏/报错                |

**后端验证：**

| #    | 检查项       | 检查内容                                             |
| ---- | ------------ | ---------------------------------------------------- |
| T-04 | 接口可用性   | mock 数据验证改动涉及的所有接口，保证不报错           |
| T-05 | 边界覆盖     | 覆盖空值、超长、非法输入等边界场景                    |
| T-06 | 返回值正确性 | 返回值结构和状态码符合 DES 定义                       |

## 4. .project 目录结构

```
.project/
  context.md                    ← 每次对话必读，AI 自动维护
  specs/
    master/
      index.md                  ← 模块状态总览，标准通道第一步读取
      requirements/
        REQ-{xx}-{name}.md      ← 需求规格，变更前确认，变更后更新
      design/
        DES-{xx}-{name}.md      ← 设计规格，设计阶段输出，编码参考
    change-log-specs.md           ← specs 变更审计日志（人工批注 + AI 级联，全部留痕）
    rules/
      project-custom.md         ← 项目铁律 + 技术栈 + 项目规范 + 外部依赖
      unit-test-base.md         ← [可选] 开发自测规则（Step 6 读取）
  changelog/                    ← 每次标准/快速通道变更生成独立文件
```
