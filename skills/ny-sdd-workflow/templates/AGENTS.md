# SDD Workflow — AI 执行规则 v3.4

> **最高指令**：严禁在未确认变更通道的情况下直接编写或修改生产代码。每次回复优先使用中文。

---

## 0. 每次对话开始

**第一步**：AI 主动询问项目类型：

```
【项目确认】
请问这是新项目还是旧项目？
  A. 新项目（请提供 PRD）
  B. 旧项目 — 新增需求
  C. 旧项目 — Bug 修复
  D. 旧项目 — 技术优化
```

**第二步**：目录骨架初始化（首次对话时创建，已存在则跳过）：

```
.docs/prd/       .docs/tech/
.project/specs/master/requirements/
.project/specs/master/design/
.project/specs/rules/
.project/specs/rules/patterns/
.project/changelog/
.outdocs/
.agents/skills/
```

**第三步**：读取以下文件恢复状态（均不存在则跳过，进入 §1 初始化）：

- `.project/context.md` — 恢复工作进度
- `.project/task.md` — 恢复子任务进度（如存在）
- `.project/specs/rules/project-profile.md` — 恢复项目认知（技术栈/构建方式/架构/铁律/规范）

**类型映射**（贯穿后续所有阶段）：

| §0 选项 | 任务类型 | 对应 §2.1 模板 |
| -------- | -------- | --------------- |
| A / B    | feature  | 新增需求        |
| C        | bug      | Bug 修复        |
| D        | refactor | 技术优化        |

## 0.5 写码门禁（Code Gate）

> AI 每次准备写/改生产代码前，必须自检以下条件，**全部 ✅ 才可执行**：

| # | 检查项                                              | 未通过处理                     |
| - | --------------------------------------------------- | ------------------------------ |
| 1 | `.project/` 已初始化                              | → 先完成图一                  |
| 2 | 当前功能有 REQ + DES 且 `review-status: approved` | → 先完成图二 §2.1-§2.4      |
| 3 | 阻塞依赖已就绪（用户明确说"跳过"的视为就绪）        | → 等待用户提供或确认跳过      |
| 4 | 技术栈对应的编码/审计 Skill 已安装                  | → 按 §1.3 Skill 执行流程安装 |
| 5 | task.md 中当前模块有 pending 子任务                 | → 定位下一个 pending 子任务   |
| 6 | 本次变更与项目铁律无冲突                            | → 列冲突点 + 替代方案，触发 §0.6 停车信号 |

**检查项 #4 Skill 安装检查规则**：根据 project-profile.md「技术栈」声明，检查以下目录是否存在：

- 技术栈含前端 → `.agents/skills/vant-vue3/`
- 技术栈含 Java → `.agents/skills/java-coding/`
- 所有项目 → `.agents/skills/code-audit/`
- 任一缺失 → 按 §1.3 Skill 执行流程安装，安装失败询问用户（手动安装 / 确认兜底）

**检查项 #5 Task 定位规则**：读取 `.project/task.md`，定位当前模块的下一个 pending 子任务，输出：

```
【当前任务】{ID} [{类型}] {描述}
【模块进度】{done}/{总数}
```

- task.md 不存在（bug 类型无 task.md；refactor 类型用户选择不创建时无 task.md）→ 跳过此检查项
- 当前模块无 pending 子任务 → 进入 §2.8 归档

**快速通道豁免**：仅样式/文案的快速通道任务，检查项 #2 简化为：REQ + DES 可各简化为一句话描述，免评审。AI 在进入 Step 0 前，在对应 REQ + DES 文件中追加一句话记录（文件不存在则新建）。检查项 #5 跳过（快速通道不更新 task.md，样式/文案改动不对应独立子任务）。

- 任一 ❌ → **停止**，输出缺失项，等待用户指示
- **禁止将"继续"、"开始"等模糊指令默认理解为"直接写代码"**，应先执行门禁自检

## 0.6 停车信号

以下情况必须暂停，向用户报告并等待指示：

| 信号 | 处理 |
|------|------|
| 路由不确定 | 说明两种判断及理由，请用户选择 |
| 影响面超预期 | 列出影响范围，建议升级通道 |
| 审计循环超 3 次未通过 | 列出未通过项，请用户判断 |
| Spec/DES 发现矛盾 | 指出矛盾点，请用户裁决 |
| 自愈循环超 5 次 | 报告错误模式，建议检查 Spec |
| 铁律冲突 | 列冲突点 + 替代方案，请用户确认 |
| 本文件未覆盖的场景 | 描述场景 + 建议处理方式，请用户确认 |

**停车不是失败，是负责任。** AI 不应在不确定的情况下"猜着往前走"。

## 1. 图一：项目启动

### 1.1 新项目流程

```
用户提供 PRD
  → AI 读取 PRD，理解业务需求
  → AI 执行 PRD 审计（必须输出 Skill 执行日志）：
    1. 按 §1.3 Skill 执行流程调用 /prd-audit Skill，输出执行日志
    2. Skill 安装失败 → 询问用户（手动安装 / 使用 §3.5.1 PRD 审计标准兜底）
    3. 兜底标准见 §3.5.1（6 维度 + P0 必须解决）
    4. P0 必须解决后才继续
  → AI 从 PRD 提取业务流程，写入 .outdocs/project-overview.md 的「二、业务架构」「三、核心业务流程」部分（新项目不写入 project-profile.md，待需求澄清完善后再沉淀）
  → AI 扫描 .docs/ 目录（如存在），按 §1.4 技术文档规则处理：
    - 扫描到的信息（技术栈/外部依赖/数据库等）直接采用，后续询问时跳过已知项
    - .docs/ 不存在或为空 → 询问用户是否现在提供技术文档：
      ```
      【技术文档确认】
      是否有技术文档需要提供？（API文档/建表脚本/中间件配置等）
        A. 现在提供（请放入 .docs/tech/ 目录）
        B. 稍后提供
        C. 没有技术文档
      ```
  → AI 询问技术栈（跳过 .docs/ 已覆盖的项）→ 写入 project-profile.md「技术栈」区块
  → AI 询问外部依赖（跳过 .docs/ 已覆盖的项）→ 写入 project-profile.md「外部依赖」区块
  → .docs/ 已覆盖的外部依赖 → 直接从文档提取，无需用户重复提供
  → AI 初始化 .project/ 文件（目录骨架已在 §0 创建）
  → AI 初始化代码结构（脚手架，必须输出 Skill 执行日志）：
    1. 查 §1.3 Skill 路由表，按技术栈匹配对应 Skill
    2. 后端 Java → 按 §1.3 Skill 执行流程调用 java-project-creator，输出执行日志
    3. 前端 → 按 §1.3 Skill 执行流程调用 wap-project-creator，输出执行日志
    4. Skill 安装失败 → 询问用户（手动安装 / 使用框架官方 CLI 兜底）
  → 安装依赖，确认能跑起来
  → AI 填写 project-profile.md 新增区块：
    按技术栈分别写入 project-profile.md 对应区域（前端 → 二、前端；后端 → 三、后端）：
    1. 「构建与运行」：从 package.json / pom.xml / Makefile 提取构建、运行、测试命令
    2. 「项目架构」：从脚手架结构提取架构模式、入口文件、路由注册方式、数据库访问模式
    3. 「内部公共能力」：扫描脚手架自带的公共工具/组件/中间件/常量/通用模式（新项目内容较少，随开发逐步追加）
  → 进入 §2.0 任务拆分（全量模块排序 + 全量模块子任务拆分）
  → 进入图二业务开发循环
```

**技术栈询问格式**：

```
【技术栈确认】
请告诉我本项目的技术栈：
  1. 前端框架？（如 Vue3 / React）
  2. 后端框架？（如 SpringBoot / Node.js）
  3. 数据库？（如 MySQL / PostgreSQL）
```

**外部依赖询问格式**：

```
【外部依赖确认】
请告诉我需要调用的外部服务：
  1. 第三方 API？（如微信支付、地图服务）
  2. 公司内部其他服务接口？
  3. 指定的前端组件库或工具库？
```

### 1.2 旧项目流程

**AI 自动执行**（读代码得出，无需人工提供）：

- 扫描 .docs/ 目录（如存在），按 §1.4 技术文档规则处理；不存在或为空 → 询问用户是否提供
- 项目结构理解 → 按前后端分别写入 project-profile.md 对应区域的「项目架构」区块（替代单独输出项目摘要，持久化保存）
- 技术栈识别（.docs/ 已覆盖的直接采用）→ 写入 project-profile.md「一、项目级 > 技术栈」区块
- 现有规范识别（命名约定 / 目录结构 / 代码风格 / commit 规范）→ 按前后端分别写入对应区域的「项目规范」区块，人工确认
- 外部依赖梳理（.docs/ 已覆盖的直接采用）→ 记录到 project-profile.md「外部依赖」区块
- 数据库结构梳理（.docs/ 已覆盖的以文档为准，否则从代码反推）
- 接口文档梳理（.docs/ 已覆盖的以文档为准，否则从代码反推）
- 构建方式识别（从 package.json / pom.xml / Makefile / Dockerfile 提取）→ 按前后端分别写入对应区域的「构建与运行」区块
- 内部公共能力扫描 → 按前后端分别写入对应区域的「内部公共能力」区块，人工确认
  扫描目录与内容：
  | 扫描目录                           | 提取什么                                              |
  | ---------------------------------- | ----------------------------------------------------- |
  | utils/、common/、lib/              | 公共工具与封装（HTTP/日期/DTO转换/分页）              |
  | config/、infrastructure/           | 基础设施封装（Redis/MQ/OSS/ES/定时任务）              |
  | components/（前端）                | 公共组件 + i18n 方案                                  |
  | composables/、hooks/（前端）       | Composables/Hooks（共享逻辑封装）                     |
  | router/（守卫部分，前端）          | 路由守卫（登录校验/权限过滤）                         |
  | directives/（前端）                | 自定义指令（v-permission 等）                         |
  | styles/、assets/css/（前端）       | 样式体系（CSS 变量/主题/公共样式）                    |
  | middleware/、interceptor/、filter/ | 全局中间件/拦截器                                     |
  | aspect/、aop/                      | AOP 切面（操作日志/性能监控/数据权限）                |
  | mapper/、service/（基类）          | 数据库通用封装（BaseMapper/自动填充/逻辑删除/多租户） |
  | security/、auth/                   | 安全/认证工具（JWT/加解密）                           |
  | constants/、enums/                 | 常量与枚举（错误码/状态枚举）                         |
  | entity/、model/、migration/        | 数据模型概览（实体/关系/字段约束）                    |
  | store/、stores/（前端）            | 全局状态管理                                          |
  | controller/、service/ 返回格式     | 项目内通用模式（返回格式/参数校验/日志方式）          |

  **记录格式要求**：每项公共能力写入 profile 时必须包含三要素——路径、用法、禁止：
  ```
  - {能力名}：`{文件路径}`
    用法：`{import 语句}; {调用示例}`
    禁止：{不允许的替代做法}
  ```
  示例：
  ```
  - HTTP 请求：`src/utils/request.ts`
    用法：`import { instance } from '@/utils/request'; instance.post<T>(url, params)`
    禁止：直接 import axios 或 fetch
  ```

- 模块架构模式扫描 → 写入 `.project/specs/rules/patterns/` 独立文件 + project-profile.md「模块架构索引」区块，人工确认
  扫描范围：
  - 前端：从路由配置识别核心页面（一级路由 + 包含复杂交互/独立数据流的页面）
  - 后端：识别核心处理管线（如 Agent 体系、消息处理链、工作流引擎等有固定注册和扩展模式的模块）
  扫描方法：
  对每个核心页面/模块，追踪其数据流和处理管线：
     | 追踪维度     | 前端                                           | 后端                                             |
     | ------------ | ---------------------------------------------- | ------------------------------------------------ |
     | 数据驱动方式 | 核心数据源是什么？如何注入组件？               | 核心输入从哪来？如何传递到处理层？               |
     | 渲染/处理管线 | 数据→UI 经过几层转换？有无路由/映射表/工厂？  | 请求→响应经过几层？有无注册表/工厂/分发器？     |
     | 组件/模块组装 | 组件间通过 props/emit/provide-inject/store？   | 模块间通过接口/事件/依赖注入/消息？             |
     | 新增同类内容 | 新增一个同类元素需要改哪几个文件？注册到哪？   | 新增一个同类处理器需要改哪几个文件？注册到哪？   |
  每个核心页面/模块生成独立文件 `.project/specs/rules/patterns/{name}.md`，内容：
     ```
     # {页面/模块名}架构模式
     ## 数据驱动
     {核心数据源及注入方式}
     ## 渲染/处理管线
     {数据到 UI/响应 的完整链路}
     ## 新增同类内容需改文件
     1. {文件路径} — {改什么}
     2. ...
     ## 禁止
     - {该页面/模块的架构红线}
     ```
  在 project-profile.md 对应区域的「模块架构索引」写入索引行（每页面/模块一行，指向独立文件）

- 业务架构与流程提取 → 写入 project-profile.md「业务架构」「业务流程」区块，人工确认
  提取来源与内容：
  | 信息源            | 提取什么               |
  | ----------------- | ---------------------- |
  | 路由/控制器       | API 全貌、模块划分     |
  | 数据库模型/实体类 | 业务实体关系、数据流向 |
  | 服务层/业务逻辑   | 核心流程、状态流转     |
  | 中间件/拦截器     | 权限模型、全局处理     |
  | 前端路由/页面     | 用户操作路径           |
  | .docs/（如有）    | 已有文档补充佐证       |

**人工确认**（AI 无法自行判断，必须问）：

- 危险区域确认（核心逻辑 / 遗留代码 / 不能动的部分）
- 分支策略确认（开发分支 / review 流程 / 合并策略）
- 部署环境确认（dev / staging / prod / CI/CD 流程）
- 敏感配置确认（密钥 / 证书 / .env 处理方式）
- 性能和业务约束（瓶颈 / 特殊要求）
- 团队协作信息（维护者 / 并行开发 / 提交规范）
- 基线确认（核心功能是否正常 / 已知 bug / 进行中分支）

**完成后**：

```
AI 生成 .project/ 目录（汇总所有信息）
  → 安装依赖，确认能跑起来
  → 生成 .outdocs/project-overview.md（按输出结构模板，从 project-profile.md 三区 + index.md + 实际代码结构汇总，详见 §4.1）
  → feature 类型 → 如用户提供了 PRD，先执行全量 PRD 审计（与 §1.1 一致）：
      1. 按 §1.3 Skill 执行流程调用 /prd-audit Skill，输出执行日志
      2. Skill 不可用 → 按 §3.5.1 PRD 审计标准兜底
      3. P0 问题必须解决后才继续
    → AI 评估需求涉及几个模块：
      - 单模块 → 进入 §2.0 任务拆分（单模块拆分 + 写入 task.md，跳过排序）
      - 多模块（2+）→ 进入 §2.0 任务拆分（全量模块排序 + 全量拆分，与新项目一致）
  → bug / refactor 类型 → 跳过 §2.0，直接进入 §2.1
  → 进入图二业务开发循环
```

### 1.2.5 project-profile.md 模板

项目铁律由团队人工填写，AI 不自动生成。其余区块由 AI 在初始化时自动填写。

**完整模板：**

```markdown

## 项目铁律（Constitution）
> 以下约定 AI 绝对不能违反，无论任何情况。违反时必须立即停止并报告。

### 技术约束
- [ ] {例：必须使用 Vue3，禁止使用 Vue2}
- [ ] {例：所有接口调用必须封装在 service 层，禁止在组件里直接调用}

### 代码规范
- [ ] {例：代码注释必须使用中文}
- [ ] {例：所有异步操作必须有错误处理}

### 业务约束
- [ ] {例：涉及金额的字段必须使用 Decimal，禁止使用 Float}
- [ ] {例：用户数据操作必须有操作日志}

## 技术栈
- frontend: {vue3 / react / ...}
- backend: {java-springboot / go / ...}
- database: {mysql / postgresql / ...}

## 外部依赖（API 学习记录）
- ...

## 原型图索引（AI 从 .docs/prd/ 图片解析，如有）
> 设计和编码时必须以原型图为视觉基准

| 文件 | 对应页面/功能 | 核心内容摘要 |
| --- | --- | --- |
| {.docs/prd/文件名} | {页面名} | {布局结构 + 关键组件 + 交互要点} |

| --- | --- | --- | --- |
| {模块名} | {职责描述} | {实体列表} | {数量} |

### 模块依赖关系
{模块A} → {模块B}（{依赖原因}）

### 角色权限总览

### 1.3 Skill 路由表

公司级规范以 Skill 形式封装，AI 在对应阶段必须自动安装并调用。

**Skill 仓库（默认）**：`https://git.nykjsrv.cn/ai-coding/skills.git`
**Skill 仓库（vant-vue3）**：`https://github.com/teachingai/full-stack-skills`
**安装命令**：`npx skills add {仓库地址} --skill {skill名称} --yes`

| 阶段           | 触发时机                     | Skill 路由            | skill 名称           | 仓库               |
| -------------- | ---------------------------- | --------------------- | -------------------- | ------------------ |
| PRD 审计       | §1.1 / §2.1                | /prd-audit            | prd-audit            | 默认               |
| 后端项目初始化 | §1.1 脚手架创建（仅新项目） | /java-project-creator | java-project-creator | 默认               |
| 前端项目初始化 | §1.1 脚手架创建（仅新项目） | /wap-project-creator  | wap-project-creator  | 默认               |
| 前端编码       | §3 Step 4 前端改动          | /frontend-coding      | vant-vue3            | vant-vue3 专用仓库 |
| 后端编码       | §3 Step 4 后端改动          | /java-coding          | java-coding          | 默认               |
| 代码审计       | §3 Step 5                   | /code-audit           | code-audit           | 默认               |

**Skill 执行流程**（所有触发点必须统一严格遵守，不可跳过）：

```
1. 检查项目目录 .agents/skills/{skill名称}/ 是否已存在
   → 已存在 → 直接进入第 3 步（调用）
   → 不存在 → 进入第 2 步（安装）
2. 安装：根据路由表中的「仓库」列选择对应仓库地址，执行 npx skills add {仓库地址} --skill {skill名称} --yes
   → 安装成功 → 进入第 3 步
   → 安装失败 → 询问用户：
     A. 手动安装（用户自行执行命令后告知 AI）
     B. 使用内置默认兜底
     用户确认后继续
3. 调用：读取 Skill 内容，按其规范执行
4. 输出执行日志（必须，未输出视为未执行）
5. 更新状态（§3 编码阶段的 Skill）：更新 index.md 对应模块的 skill-status
```

**Skill 执行日志格式**（每次执行 Skill 流程后必须输出，禁止省略）：

```
【Skill 执行日志】
  阶段: {触发阶段，如 §1.1 PRD审计 / §3 Step4 前端编码}
  Skill: {skill名称}
  检查: .agents/skills/{skill名称}/ → {已存在 / 不存在}
  安装: {跳过(已存在) / 安装成功 / 安装失败}
  执行: {调用Skill / 内置兜底({兜底方式})}
```

**匹配规则**：

- 根据 project-profile.md「技术栈」声明 + 当前改动文件类型，匹配对应 Skill
- Skill 安装失败 → 询问用户（手动安装 / 使用内置默认兜底），由用户确认后继续
- **项目初始化（仅新项目 §1.1）**：技术栈含 Java → java-project-creator；技术栈含前端 → wap-project-creator；旧项目（§1.2）已有代码结构，不触发
- **编码阶段**：前端改动调用 /frontend-coding，后端改动调用 /java-coding（§3 Step 4）
- **代码审计**：前后端统一调用 /code-audit（§3 Step 5）

**融合规则**（project-profile.md 与 Skill 的关系）：

1. 读取 project-profile.md（铁律 + 项目规范）
2. 调用 Skill 获取公司级规范
3. 冲突项 → 以 project-profile.md 为准
4. project-profile.md 未提及 → 用 Skill 规范
5. 两者都未覆盖 → 用内置默认

### 1.4 技术文档（.docs/）

用户将文档放入项目根目录 `.docs/` 的对应子目录：

- `.docs/prd/` — PRD 需求文档
- `.docs/tech/` — 技术文档（API 文档、建表脚本、中间件配置、对接手册等）

文件格式和命名不限，可以多个文件也可以全部放一个文件。

**AI 处理规则**：

**何时扫描**：

- §1.1 / §1.2 项目初始化时 → `.docs/` 存在则自动扫描全部文件
- 用户说"文档更新了" / ".docs 有新文件" / "我加了文档" → 重新扫描
- §2.2 方案设计时 → 读取当前模块相关内容
- §3 Step 4 编码时 → 读取当前模块相关内容

**怎么扫描**：

1. 按子目录分类处理：
   - `.docs/prd/` → 作为需求输入，走 §1.1 PRD 读取或 §2.4.5 触发源 B（PRD 变更）
   - `.docs/prd/` 中的图片文件（png/jpg/jpeg/webp/svg）→ 作为**原型图/UI 设计稿**处理（见下方原型图处理规则）
   - `.docs/tech/` → 走技术文档流程（识别 API 接口 / 数据库结构 / 中间件配置 / 其他）
   - `.docs/` 根下的文件（兼容旧结构）→ AI 按内容自动判断：含 PRD 特征（需求描述/用户故事/功能清单）走 PRD 流程，否则走技术文档流程，同时提示用户归入对应子目录
2. 技术文档在 project-profile.md「外部依赖」区块建立索引（内容摘要 + 来源文件路径）

**原型图处理规则**：

扫描到 `.docs/prd/` 中的图片文件时：

1. **解析**：AI 读取图片，深度分析以下内容：
   - 页面布局结构（区域划分、层级关系）
   - UI 组件识别（按钮/表单/列表/卡片/弹窗等，含组件类型和状态）
   - 交互线索（箭头/流程图/标注/页面间跳转关系）
   - 视觉规范（颜色/字号/间距/圆角等，如可识别）
   - 数据字段（表单字段名/列表列名/卡片信息项）

2. **建索引**：写入 project-profile.md「原型图索引」区块：
   ```
   ## 原型图索引
   > 以下从 .docs/prd/ 图片解析，设计和编码时必须以原型图为视觉基准

   | 文件 | 对应页面/功能 | 核心内容摘要 |
   | --- | --- | --- |
   | .docs/prd/{文件名} | {页面名} | {布局结构 + 关键组件 + 交互要点} |
   ```

3. **贯穿使用**（各阶段引用原型图的规则）：
   - **§2.1 需求分析**：读取对应原型图，从中提取 UI 需求（字段、交互、状态）补充到 REQ 验收标准
   - **§2.2 方案设计**：DES 的前端部分必须参照原型图设计，布局/组件选择/交互流程与原型图对齐，偏离处须在 DES 中标注原因
   - **§2.3 原型生成**：有用户提供的原型图时，以其为基准生成高保真原型，不另起炉灶；无原型图时才从 DES 生成
   - **§3 Step 4 前端编码**：加载当前任务对应的原型图作为视觉参照，页面结构和组件必须与原型图一致


**怎么用**：

- 涉及外部 API → 接口路径、参数、响应格式、认证方式**必须以 `.docs/` 中的文档为准**，禁止编造
- 涉及数据库 → 表结构、字段类型、索引**必须以 `.docs/` 中的文档为准**

| ------------------- | --------------------------------- | ----------------------------- |
| `pending`         | 未执行任何 Skill                  | Step 4 必须执行 Skill 流程    |
| `coding:done`     | 编码 Skill 已执行                 | Step 4 跳过 Skill，直接编码   |
| `coding:fallback` | 编码 Skill 安装失败，用户确认兜底 | Step 4 使用内置规约 C-01~C-10 |
| `audit:done`      | 审计 Skill 已执行                 | Step 5 跳过 Skill，直接审计   |
| `audit:fallback`  | 审计 Skill 安装失败，用户确认兜底 | Step 5 使用内置规约 S-01~S-06 |
| `all:done`        | 编码 + 审计 Skill 均已执行        | Step 4/5 跳过 Skill 流程      |

**Step 2：文档学习（按需）**

先查 project-profile.md 是否已覆盖当前任务涉及的内容：一、项目级的「外部依赖」，以及按任务类型查询对应区域（前端 → 二、前端；后端 → 三、后端）的「构建与运行」「项目架构」「内部公共能力」，已覆盖则跳过。未覆盖时，优先从 `.docs/` 中查找（按 §1.4 规则）。`.docs/` 也未覆盖且涉及未记录的 API / 库时触发：

```
【文档学习确认】
本次任务涉及 {API/库名}，是否有文档需要我先学习？
支持：A. 在线链接  B. PDF  C. 粘贴内容  D. 放入 .docs/tech/ 目录
```

学习后输出摘要，反填 `project-profile.md`「外部依赖」区块。

**Step 3：评估影响面**

- 判断本次改动波及哪些模块
- 识别潜在风险点
- 按以下维度输出影响面清单：

| 维度     | 检查内容                                                        |
| -------- | --------------------------------------------------------------- |
| 功能关联 | 其他模块是否依赖或调用被修改的功能                              |
| 数据关联 | 是否涉及共享数据模型/表/缓存                                    |
| 业务流程 | 是否影响 project-profile.md「业务流程」中的现有流程（仅旧项目） |
| 页面/模块架构 | 是否影响 patterns/*.md 中已有页面/模块的渲染管线/处理链       |
| 安全影响 | 是否涉及认证/授权/脱敏/校验                                     |
| 性能影响 | 是否影响查询性能/并发/内存                                      |

输出格式：
```
【影响面评估】
  - 功能关联：{无 / 影响 XXX 模块的 XXX 功能}
  - 数据关联：{无 / 涉及 XXX 表}
  - 业务流程：{无 / 影响 XXX 流程}
  - 页面/模块架构：{无 / 影响 XXX 的渲染管线/处理链}
  - 安全影响：{无 / 涉及 XXX}
  - 性能影响：{无 / 可能影响 XXX}
  结论：{影响范围可控，继续 / 影响面超出当前通道预期，触发 §0.6 停车信号}
```

**Step 4：执行代码变更**

- **task.md 联动（feature 类型，或 refactor 有 task.md 时）**：编码开始前，将当前子任务 status 更新为 `in-progress`
- 符合 `.project/specs/` 规范
- 检查 index.md 当前模块的 skill-status：
  - `pending` → 按 §1.3 Skill 执行流程调用编码 Skill（前端 /frontend-coding，后端 /java-coding），**必须输出 Skill 执行日志**，完成后更新 skill-status 为 `coding:done` 或 `coding:fallback`
  - `coding:done` / `coding:fallback` / `all:done` → Skill 已执行，跳过 Skill 流程
- 按融合规则与 project-profile.md 融合，Skill 不可用则由用户确认是否使用内置规约
- 编码时必须遵循 Step 0 加载的编码上下文（项目规范 + 内部公共能力 + 架构约束），复用已有能力，禁止重复封装
- 编码过程中新增公共工具/组件/枚举 → 按任务类型追加记录到 project-profile.md 对应区域（前端 → 二、前端；后端 → 三、后端）的「内部公共能力」对应子节
- 编码过程中为某页面/模块引入新的架构模式（如新的渲染管线、处理链、注册机制） → 补写 `.project/specs/rules/patterns/{name}.md` + 更新 project-profile.md「模块架构索引」
- 逐项遵守编码规约（融合优先级：project-profile.md > Skill 规范 > C-01~C-10）：

| #    | 规约             | 要求                                          |
| ---- | ---------------- | --------------------------------------------- |
| C-01 | 异常路径完整     | 每个 try-catch 有错误分类、用户提示、日志记录 |
| C-02 | 输入校验双重防线 | 前端校验+后端校验，后端不可省                 |
| C-03 | 幂等与并发安全   | 写操作幂等（Token/唯一键），并发敏感加锁      |
| C-04 | 配置外部化       | 所有环境配置走环境变量，零硬编码              |
| C-05 | 结构化日志       | 关键路径有结构化日志，错误日志含上下文        |
| C-06 | 超时与降级       | 所有外部调用设超时，超时有降级方案            |
| C-07 | 数据边界处理     | null/空数组/超长文本/特殊字符有防御           |
| C-08 | 安全基线         | SQL参数化、XSS转义、敏感数据不进日志          |
| C-09 | 错误码体系       | 业务错误有明确错误码                          |
| C-10 | 资源释放         | 连接/句柄/定时器用完必释放                    |

**Step 5：代码审计**

- 检查 index.md 当前模块的 skill-status：
  - `coding:done` / `coding:fallback` → 按 §1.3 Skill 执行流程调用 /code-audit Skill，**必须输出 Skill 执行日志**，完成后更新 skill-status 为 `all:done`（或 `audit:fallback`）
  - `all:done` → Skill 已执行，跳过 Skill 流程
  - `pending` → Step 4 未执行，回退到 Step 4
- 按融合规则与 project-profile.md 融合，Skill 不可用则由用户确认是否使用内置规约
- 融合优先级：project-profile.md > Skill 规范 > §3.5.4 代码审计标准
- 按 §3.5.4 S-01~S-06 逐项检查，不通过项当场修复，修复后重新检查该项，直到全部通过。单项修复超 3 次仍未通过 → 触发 §0.6 停车信号（列出未通过项，请用户判断）

- S-04 性能回归：✅
- S-05 权限校验：✅
- S-06 能力复用：✅
```

## 3.5 质量标准体系

各阶段产出物的审计标准统一定义如下，具体执行时机见对应步骤引用。

### 3.5.1 PRD 审计标准

| 维度 | 检查内容 |
|------|---------|
| 完整性 | 正常 + 异常流程是否完整 |
| 一致性 | 与现有功能/数据模型是否冲突 |
| 可行性 | 现有架构是否支撑 |
| 边界 | 做什么/不做什么是否明确 |
| 优先级 | P0/P1/P2 是否明确 |
| 安全合规 | 权限/数据安全/隐私是否覆盖 |

P0 问题必须解决后才继续。

### 3.5.2 REQ 审计标准

| 维度 | 检查内容 |
|------|---------|
| 验收标准具体性 | 每条验收标准是否可验证，无"正常""合理""等情况"等模糊词 |
| 边界明确性 | 做什么/不做什么是否清晰，无开放式范围 |
| 与 PRD 一致 | 是否忠实于 PRD 对应部分，无偏离无遗漏 |
| 异常场景 | 异常/边界场景是否至少列出，不留空白 |

### 3.5.3 DES 审计标准

| 维度 | 检查内容 |
|------|---------|
| 维度覆盖 | feature 类型的 5 维度（数据模型/API/核心流程/状态管理/权限）是否涉及项均已覆盖 |
| 与 REQ 一致 | 设计方案是否覆盖 REQ 全部验收标准，无遗漏无偏离 |
| 可编码性 | 读完 DES 即可编码，无需猜测任何实现细节 |
| 异常穷举 | 异常路径是否完整定义，不留"等情况" |
| 架构一致 | 方案是否与 project-profile.md 的架构模式一致；复用能力清单是否正确匹配了 profile 中已有的内部公共能力，无遗漏复用、无重复造轮子 |

### 3.5.4 代码审计标准（S-01 ~ S-06）

| # | 审计项 | 检查内容 |
|---|--------|---------|
| S-01 | 空值防御 | null/undefined/空数组是否有防御 |
| S-02 | 前端输入信任 | 后端是否独立校验，不信任前端 |
| S-03 | Spec 一致性 | 实现是否与 REQ + DES 一致 |
| S-04 | 性能回归 | 是否引入慢查询/大循环/内存问题 |
| S-05 | 权限校验 | 接口权限是否正确配置 |
| S-06 | 能力复用 | 是否按 DES「复用能力」清单使用了 profile 中已有的内部公共能力，无重复封装（如自行封装 axios 而非用项目 request 工具） |

### 3.5.5 开发自测标准（T-01 ~ T-06）

**前端验证：**

| # | 检查项 | 检查内容 |
|---|--------|---------|
| T-01 | 交互逻辑 | 改动点涉及的每个页面中，相关按钮的交互逻辑是否正常 |
| T-02 | 页面跳转 | 改动涉及的页面跳转路径是否正确 |
| T-03 | 页面渲染 | 改动涉及的页面是否正常渲染，无白屏/报错 |

**后端验证：**

| # | 检查项 | 检查内容 |
|---|--------|---------|
| T-04 | 接口可用性 | mock 数据验证改动涉及的所有接口，保证不报错 |
| T-05 | 边界覆盖 | 覆盖空值、超长、非法输入等边界场景 |
| T-06 | 返回值正确性 | 返回值结构和状态码符合 DES 定义 |
