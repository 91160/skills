# Reverse Scan — 深度业务代码扫描流程

> 本文件定义 reverse-scan Skill 的完整执行流程。由 SKILL.md 引用，AI 读取后逐步执行。

---

## 0. 执行模式与强制暂停规则

### 0.1 执行模式

**执行模式参数：** `mode: auto | interactive`，**默认 `auto`**。

- 调用方/操作者明确说"交互扫描"、"每步确认"、"interactive" → 切为 `interactive`
- 调用方/操作者说"直接扫完"、"一把扫完"、"全自动"或未指定 → 使用 `auto`

**Phase / 审计 / 模块切换的硬约束（两种模式都适用）：**
Phase 之间、审计通过后、模块切换、Step 切换一律**不得输出"是否继续 / 要不要进入下一步"之类的提问**。AI 必须直接进入下一动作。仅当命中 §0.2 硬暂停条件（H-05 / H-06 / H-07）时才允许中断。

**审计-修复循环自动执行：** 每个 Phase 末尾的审计由 AI 自动完成。审计未通过时 AI 自动修复并重新审计。循环次数上限为 **5 次**，超过则触发 H-07 暂停。循环中间态**静默处理**：循环次数 < 5 时不输出每一轮的进度，仅在 `scan-summary.md` 中记录"某模块/某审计共 N 轮通过"。

### 0.2 暂停条件分级

**硬暂停（两种模式都必须停）**：AI 无法独断的情形，命中后立即输出报告并等待操作者。

| 编号 | 条件 | 输出格式 |
|------|------|---------|
| H-05 | 需要操作者提供代码无法反映的信息（业务规则、外部约束等） | `【硬暂停：需人工补充】` + 具体需要的信息 + 为什么代码里推断不出来 |
| H-06 | 交叉验证发现 Spec 与代码行为不等价，且 AI 修复仍不成功 | `【硬暂停：验证不通过】` + 不等价清单 + 差异描述 |
| H-07 | 某个审计-修复循环超过 5 次仍未通过 | `【硬暂停：审计循环超限】` + 哪个审计项 + 连续未通过的原因 + 建议 |

**软暂停（auto 模式下降级为 AI 自决 + 记录，interactive 模式下仍停）**：AI 在有限信息下可做出合理判断的灰色地带。

| 编号 | 条件 | auto 模式处理 | interactive 模式处理 |
|------|------|---------------|----------------------|
| H-01 | 疑似 dead code（有定义但全项目无调用） | AI 按置信度决策：高置信度直接标记"疑似废弃，保留不纳入任何模块"并记入 grey-decisions.md；低置信度累积到本 Phase 末尾批量输出 | 累积到本 Phase 末尾一次暂停 + 清单 + AI 建议 |
| H-02 | magic number / 硬编码常量，业务含义不明 | 高置信度（能从调用处/命名推出）直接填 AI 猜测并记录；低置信度累积批量输出 | 累积到本 Phase 末尾一次暂停 |
| H-03 | 代码逻辑与注释/文档矛盾 | AI 以代码为准（代码是唯一事实源），记录矛盾点到 grey-decisions.md，在 Phase 6 复核清单中列出 | 立即暂停 + 矛盾点 + 两版本含义 |
| H-04 | workaround 模式（静默吞异常、TODO/FIXME/HACK、特殊值绕过） | 高置信度按"技术债"归类记录，不影响 DES/REQ 生成；低置信度累积批量输出 | 累积到本 Phase 末尾一次暂停 |

**置信度判定规则：** AI 基于以下三点自判高/低置信度——① 是否能从调用方、命名、上下文推断出合理业务含义；② 错判影响面（不影响模块划分/DES 正确性 → 高；可能影响 → 低）；③ 是否存在矛盾信号。三点都不支持才判低置信度。

**grey-decisions.md 记录要求（auto 模式自决时）：** 每条自决必须记录：位置、可疑类型、AI 判断、置信度、判断依据。供 Phase 6 末尾一次性复核。

**末尾集中复核（两种模式都适用，参见 Phase 6 / Step R5.4）：** Phase 6 汇总所有软暂停自决项 + 低置信度批量项为一份"灰色决策复核清单"，由操作者一次性确认/修正。

**暂停恢复规则：** 操作者回复后，AI 记录决策到 grey-decisions.md，然后从暂停点继续执行。

**可疑标记批量处理规则：** 同类可疑标记合并记录，auto 模式下不触发中途暂停，仅 interactive 模式在 Phase 末尾合并为一次暂停。

## 1. 执行流程总览

```
Phase 1: 基础设施增强（补全调用方基础扫描未覆盖的深度信息）
         ↓ 审计 A1 通过
Phase 2: 底层知识提取（逐文件逐函数，生成知识卡片 + 调用关系图）
         ↓ 审计 A2 通过
Phase 3: 知识聚合与模块发现（从知识卡片中涌现模块边界）
         ↓ 审计 A3 通过
Phase 4: 已有模块 Spec 生成（逐模块生成 DES + REQ，产出到 reverse-scan/specs/）
         ↓ 每个模块 A4-DES + A4-REQ 通过
Phase 5: 项目级文档整合（生成合并就绪文件：profile-patch + overview + api-doc）
         ↓ 审计 A5 通过
Phase 6: 全局交叉验证 + 收尾
         ↓ 审计 A6 通过
完成 → 输出扫描报告 → 控制权交还调用方
```

## Phase 1: 基础设施增强

> **目标：** 在调用方已识别的技术栈、外部依赖、前后端规范基础上，补全两项深度信息：完整数据库结构和细化的公共能力扫描。
>
> **明确不做的事：** 模块划分、业务架构、业务流程。这些必须等 Phase 2 底层知识提取完毕后才能做。

### Step 1.1: 数据库结构深度提取

检查调用方是否已产出数据库结构信息：

- `.docs/tech/` 中有建表脚本或数据库文档 → 以文档为准，补全代码中有但文档中未覆盖的表
- 无文档 → 从代码反推：扫描 entity / model / migration / schema / DDL 文件 + ORM 配置

产出写入 `.project/reverse-scan/db-schema.md`（模板见 card-template.md §4），要求：
- 每张表有完整字段定义（字段名 / 类型 / 约束 / 默认值 / 说明）
- 索引清单（索引名 / 字段 / 用途）
- 外键关系（本表.字段 → 目标表.字段）
- 表分类（业务表 / 配置表 / 关联表 / 系统表）

### Step 1.2: 公共能力深度扫描

在 project-profile.md 和 context 文件已有的公共能力基础上，按以下目录表做更细粒度的扫描：

| 扫描目录 | 提取什么 |
|---------|---------|
| utils/、common/、lib/ | 每个公共函数的签名 + 一句话业务语义 + 使用示例 |
| config/、infrastructure/ | 基础设施封装的配置方式 + 对外暴露的 API |
| components/（前端） | 每个公共组件的 props / events / slots |
| composables/、hooks/（前端） | 每个 hook 的参数 + 返回值 + 用途 |
| middleware/、interceptor/、filter/ | 处理逻辑 + 执行顺序 + 影响范围 |
| aspect/、aop/ | 切面逻辑 + 切入点 + 影响的方法范围 |
| constants/、enums/ | 每个常量/枚举的值 + 业务含义 |
| store/、stores/（前端） | 每个 store 的 state / getters / actions |

产出：
- 公共能力细节记录到知识卡片中（归入公共能力，不归入业务模块），供主工作流在合并阶段按需提取到 context 文件
- 不直接写入 project-profile.md 或 context 文件

| # | 检查项 | 通过条件 |
|---|--------|---------|
| A1-1 | 数据库结构完整 | db-schema.md 包含所有表的定义，每张表有字段名/类型/约束 |
| A1-2 | 数据库来源标注 | 每张表标注了来源（文档 / 代码反推 / 混合） |
| A1-3 | 公共能力细化 | Step 1.2 表中存在的目录已扫描，产出已记录到知识卡片中 |
| A1-4 | 基础设施不含业务猜测 | project-profile.md 中「业务架构」「业务流程」区块为空或标注"待 Phase 3 后填充" |

**审计流程：** AI 逐条自检 → 全部通过进入 Phase 2 / 未通过修复后重新审计（上限 5 次）。

## Phase 2: 底层知识提取

> **目标：** 逐文件、逐类、逐函数地阅读业务代码，为每个代码单元生成知识卡片。这是整个扫描流程的事实基础。
>
> **关键原则：** 只记录事实，不做推测。"这个函数做了什么"是事实，"这个函数属于哪个模块"是推测——推测留给 Phase 3。

### Step R1.1: 确定扫描范围

从项目目录中识别所有业务代码文件，**排除以下内容**：
- Phase 1 已扫描的公共能力代码（utils/、common/、config/ 等，但这些已有知识卡片）
- 第三方依赖（node_modules/、.m2/、vendor/ 等）
- 构建产物（dist/、build/、target/ 等）
- 测试代码（暂存，Phase 6 验证时参考）
- 配置文件（Phase 1 已处理）

产出：待扫描的业务代码文件清单。

### Step R1.2: 逐文件知识提取

对每个业务代码文件，按知识卡片模板（card-template.md §1）提取信息。

**提取粒度：**

**文件级：**
- 文件路径、文件角色、文件内的类/函数清单、关联表

**函数/方法级（核心，对每个非 trivial 的函数/方法）：**

| 维度 | 提取内容 |
|------|---------|
| 标识 | 函数签名（名称 + 参数 + 返回值） |
| 业务语义 | 用一句业务语言描述（如"根据报告ID查询解析结果"，不是"调用mapper查数据库"） |
| 输入 | 参数列表（名称/类型/约束/可选性） |
| 输出 | 返回值（类型/结构/可能的值域） |
| 核心逻辑 | 处理步骤（业务语言，非代码直译） |
| 调用了谁 | 调用的其他函数（精确到 类名.方法名） |
| 被谁调用 | 当前文件内可确定的调用者（跨文件在 Step R1.3 补充） |
| 操作的数据 | 读/写了哪些表的哪些字段 |
| 异常处理 | 异常分支 + 处理方式 |
| 状态变更 | 是否修改状态字段，从什么到什么，什么条件 |
| 权限要求 | 权限注解/检查 |
| 可疑标记 | dead code / magic number / workaround / TODO / FIXME |

**跳过的内容：**
- getter/setter、toString、hashCode/equals 等样板代码
- 私有辅助方法可归入调用它的公开方法的核心逻辑描述中

**可疑标记处理规则：**
- 发现 H-01~H-04 对应的可疑项 → 在知识卡片中标记
- auto 模式：按 §0.2 置信度规则自决并记入 grey-decisions.md，**不触发暂停**
- interactive 模式：同类可疑项累积到本 Phase 文件清单扫描完毕后，批量触发一次暂停

**前端页面额外提取（Step R1.2.1）：**

对前端页面文件，除函数级提取外，追加页面级信息（模板见 card-template.md §1 前端页面额外维度）：
- 页面路由、页面用途、组件结构、用户交互、API 调用、页面跳转

### Step R1.3: 跨文件调用关系补全

所有文件的知识卡片生成完毕后，遍历全部卡片：

1. 对每个函数的"调用了谁"列表，找到被调用函数的知识卡片，在其"被谁调用"中补充记录
2. 生成全局调用关系图 `.project/reverse-scan/call-graph.md`（模板见 card-template.md §2）
3. 识别"被多方调用的关键节点"（被 3 个以上不同文件调用的函数）
4. 识别"孤立节点"（有定义但无调用者的非入口函数）→ 标记为疑似 dead code

### 审计 A2: 底层知识提取审计

| # | 检查项 | 通过条件 |
|---|--------|---------|
| A2-1 | 文件覆盖率 | Step R1.1 确定的每个业务代码文件都有对应的知识卡片 |
| A2-2 | 函数覆盖率 | 每个非 trivial 的公开方法都在知识卡片中有记录 |
| A2-3 | 业务语义非空 | 每个函数的"业务语义"字段已填写，且不是代码直译 |
| A2-4 | 调用关系闭合 | 知识卡片中"调用了谁"引用的函数，在另一张知识卡片中存在（公共能力和第三方库调用除外） |
| A2-5 | 数据操作可追溯 | 每个"操作的数据"中引用的表，在 db-schema.md 中存在 |
| A2-6 | 可疑标记已处理 | 所有 H-01~H-04 可疑项均已处理：auto 模式下 AI 自决并记入 grey-decisions.md（含置信度和依据）；interactive 模式下已暂停获得操作者决策 |
| A2-7 | 调用关系图完整 | call-graph.md 已生成，"被多方调用的关键节点"表已填充 |

**审计流程：** AI 逐条自检 → 全部通过进入 Phase 3 / 未通过修复后重新审计（上限 5 次）。

## Phase 3: 知识聚合与模块发现

> **目标：** 从 Phase 2 的知识卡片中，通过分析调用关系和数据关系，让模块边界自然涌现。不预设模块划分，让事实说话。

### Step R2.1: 聚类分析

基于知识卡片和调用关系图，按四个维度聚类：

**3.1 数据亲和度聚类：** 操作同一组数据库表的函数倾向于属于同一个模块。
- 从所有知识卡片中提取"操作的数据"
- 将操作相同表集合的函数归为一簇

**3.2 调用亲和度聚类：** 互相调用频繁的函数倾向于属于同一个模块。
- 从 call-graph.md 分析调用密度
- 内部调用远多于外部调用的函数集合倾向于是一个模块

**3.3 API 入口聚类：** 同一个 Controller/Router 下的接口通常服务于同一个业务场景。
- 注意：仅为启发项，不是决定性依据。一个 Controller 可能混合了多个模块的接口

**3.4 前端页面聚类：** 互相跳转的页面、使用相同 API 的页面倾向于属于同一个模块。

### Step R2.2: 模块边界确定

综合四个维度，确定模块划分：

**判定规则：**
1. 数据亲和度和调用亲和度一致指向同一组函数 → 高置信度模块
2. 数据亲和度和调用亲和度冲突 → 先按数据亲和度划分，标记"边界待确认"
3. 一个函数被多个模块调用 → 不含业务逻辑（纯工具）→ 归入公共能力；含业务逻辑 → 归入数据亲和度最高的模块，其他模块标记依赖

### Step R2.3: 模块间依赖分析

基于知识卡片中的调用关系，确定模块间的依赖：
- 方向：{模块A} → {模块B}
- 原因：A 的 {函数X} 调用 B 的 {函数Y}，用于 {业务目的}

### Step R2.4: 生成模块地图

产出写入 `.project/reverse-scan/module-map.md`（模板见 card-template.md §3），包括：
- 每个模块的完整定义（职责/知识卡片/函数/表/API/页面/实体/置信度）
- 模块间依赖关系表
- 公共能力归属表
- 聚类统计
- 边界待确认清单

---

### 审计 A3: 模块发现审计

| # | 检查项 | 通过条件 |
|---|--------|---------|
| A3-1 | 知识卡片全覆盖 | 每张知识卡片要么归入某个模块，要么归入公共能力，要么标记"未归入"并说明原因 |
| A3-2 | 模块自洽性 | 每个模块内的函数之间存在调用关系或数据关系 |
| A3-3 | 模块边界不重叠 | 一个业务函数只归入一个模块（公共能力除外） |
| A3-4 | 依赖关系无环 | 模块间依赖无循环。如发现循环 → 考虑合并模块或重新划定边界 |
| A3-5 | API 全覆盖 | 代码中所有 API endpoint 都归入了某个模块 |
| A3-6 | 数据表全覆盖 | db-schema.md 中所有业务表至少被一个模块关联（纯配置表/系统表除外） |

**审计流程：** AI 逐条自检 → 全部通过进入 Phase 4 / 未通过修复后重新审计（上限 5 次）。

## Phase 4: 已有模块 Spec 生成

> **目标：** 逐模块生成 DES 和 REQ，产出到 `.project/reverse-scan/specs/`。
>
> **注意：** 这些 Spec 是已有代码的快照，仅供后续编码参考，不写入正向 Spec 体系的 `specs/master/` 和 `index.md`。

### 执行顺序

按模块间依赖的拓扑顺序（被依赖的模块排前面）逐模块执行。每个模块完整走完 Step R3.1 → 审计 A4-DES → Step R3.2 → 审计 A4-REQ 后，**自动**进入下一个模块。**不得在模块之间输出"是否继续"等提问**；审计循环中间态静默，仅在所有模块完成后统一汇总到 scan-summary.md。

### Step R3.1: 生成 DES 文件

**输入：** 该模块关联的所有知识卡片 + db-schema.md 对应子集 + 模块依赖信息

**路径：** `.project/reverse-scan/specs/design/DES-{xx}-{name}.md`（模板见 card-template.md §5）

**DES 生成规则：**

1. **从知识卡片聚合，不重新读代码。** DES 的所有内容必须可追溯到知识卡片中的某个事实。知识卡片中没有的逻辑不允许出现在 DES 中（意味着 Phase 2 遗漏了，需回补）
2. **去实现、提业务。** 知识卡片是"代码做了什么"，DES 转化为"业务要求是什么"。例如知识卡片说"调用 RedisTemplate.opsForValue().set(key, value, 30, MINUTES)"，DES 写"解析结果缓存 30 分钟"
3. **按模板组织。** 覆盖技术方案、数据模型、API 定义、核心流程、状态管理、权限矩阵、影响范围

---

### 审计 A4-DES: DES 与源代码一致性审计

> **这是整个扫描流程中最关键的审计。** DES 必须与底层代码的实际行为完全一致。

**审计方法：重新阅读该模块的源代码，逐条比对 DES 中的描述。**

注意：此处必须重新读源代码，不能只对照知识卡片。知识卡片本身可能有遗漏。

| # | 检查项 | 通过条件 |
|---|--------|---------|

## 附录 B: 关键设计说明

**B.1 为什么自底向上？**
代码是唯一事实源。自顶向下会导致 AI 从目录结构猜测业务架构，再带着偏见读代码。自底向上确保所有上层知识建立在逐行阅读的事实基础上。

**B.2 为什么需要知识卡片？**
知识卡片是"代码事实"和"Spec 描述"之间的桥梁。直接从代码生成 DES 容易变成代码直译。知识卡片强制 AI 先用业务语言理解每个函数，再从理解中聚合出 DES。

**B.3 为什么 A4-DES 审计要重读源代码？**
知识卡片可能有遗漏（Phase 2 审计只能检查格式完整，无法检查是否遗漏了某个 if 分支）。DES 从知识卡片聚合，如果卡片遗漏 DES 也会遗漏。所以 A4-DES 必须回到代码做最终校验。

**B.4 为什么 Spec 产出到 reverse-scan/specs/ 而非 specs/master/？**
路径隔离。specs/master/ + index.md 是正向 Spec 体系，只放新需求新设计。reverse-scan/specs/ 是已有代码的 Spec 快照，供后续编码参考但不混入正向体系。

**B.5 为什么 REQ/DES 标记 review-status: approved？**
逆向的源头是已在运行的代码——代码已是"被批准"的状态。标记 `source: reverse-scan` 以区分正向产出。
