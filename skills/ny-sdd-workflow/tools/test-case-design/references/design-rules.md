# 测试用例设计规则 — 完整清单

本文件定义测试用例设计方法论和 TC-01~TC-06 自审规则。供 `test-case-design` Skill 的 Step 3（生成用例）和 Step 5（自审）引用。

---

## 一、6 种设计方法论

测试用例设计是一门经典学科，本 Skill 采用 6 种成熟方法论。AI 生成用例时**必须显式使用这些方法论**，在 TC 文档"覆盖率矩阵"章节下方列出每条用例对应的方法论名。

**不使用**：正交实验法（对 AI 生成价值有限，组合爆炸时改用判定表+场景法）。

### 1. 等价类划分（Equivalence Partitioning）

**定义**：把输入域划分成若干个等价类，每个等价类中选一个代表性值进行测试，假设该类中其他值的行为与代表值一致。

**适用场景**：
- 输入字段有明确的取值范围或类型约束
- 验收标准可以按"正确/错误"或"类型"归类
- 用于**第一章 业务流程测试**、**第四章 异常与边界测试**

**划分维度**：
- **有效等价类**：合法输入（如年龄 1~150）
- **无效等价类**：非法输入（如年龄 <1、>150、非数字、空）

**示例**：
- 字段 `username`（4~20 位字母数字）
- 有效等价类：`"zhang3"`、`"UserAdmin_1"`
- 无效等价类：`""`（空）、`"abc"`（太短）、`"a".repeat(21)`（太长）、`"张三"`（中文）、`"user name"`（含空格）

**AI 使用规则**：
- 每个字段至少生成 1 条有效等价类用例 + 2 条无效等价类用例
- 生成的用例在 TC 文档中标注 `(方法论: 等价类划分)`

### 2. 边界值分析（Boundary Value Analysis）

**定义**：在有效等价类和无效等价类的**边界**选择测试值，因为错误最容易发生在边界。

**适用场景**：
- 数值字段（年龄、金额、数量、长度）
- 字符串长度限制
- 集合大小限制
- 时间范围
- 用于 **TC-03 边界值覆盖** 审计项

**覆盖维度**：

| 维度 | 必测值 |
|---|---|
| 数值字段 | `min` / `min-1`（越界低） / `min+1` / `max-1` / `max` / `max+1`（越界高） / `0` / 负数（若允许） |
| 字符串长度 | 空 `""` / 1 字符 / `maxLen-1` / `maxLen` / `maxLen+1` / 超长（比如 10000 字符） |
| 特殊字符 | 空白 / 换行 / Unicode / Emoji / HTML 标签 / SQL 关键字 / 引号 / 反斜杠 |
| 集合大小 | 空数组 `[]` / 单元素 / `maxItems-1` / `maxItems` / `maxItems+1` / 大量元素（1000+） |
| 时间 | 当前 / 过去 / 未来 / 跨年边界 / 跨月边界 / 闰年 2-29 / 时区切换时刻 |

**AI 使用规则**：
- 每个数值字段必须覆盖 min/max/越界高/越界低 四个边界
- 每个字符串字段必须覆盖空/maxLen/maxLen+1 三个边界
- 每个字段必须至少覆盖 1 类特殊字符（通常是 SQL 注入防护检查）
- 生成的用例在 TC 文档中标注 `(方法论: 边界值分析)`

### 3. 判定表（Decision Table）

**定义**：列出所有条件组合的真值表，每个组合对应一条测试用例。适合"多个条件决定多个结果"的场景。

**适用场景**：
- 规则引擎、权限判断、价格计算
- 条件数 ≥2 且条件之间有组合关系
- 用于**第一章 业务流程测试**（状态流转场景）

**示例**：优惠券使用规则

| 用户等级 | 订单金额 >= 满减阈值 | 商品在优惠品类 | → 是否可用 |
|---|---|---|---|
| 普通 | 是 | 是 | ✅ 可用 |
| 普通 | 是 | 否 | ❌ 不可用 |
| 普通 | 否 | 是 | ❌ 不可用 |
| 普通 | 否 | 否 | ❌ 不可用 |
| VIP | 是 | 是 | ✅ 可用 |
| VIP | 是 | 否 | ✅ 可用（VIP 特权） |
| ... |

每一行 → 一条 TC 用例。

**AI 使用规则**：
- 条件数 ≥2 时，必须构造判定表
- 条件数 ≥4 时，用"因子合并法"压缩组合（保留关键组合 + 各因子的代表性组合）
- 生成的用例在 TC 文档中标注 `(方法论: 判定表)`

### 4. 场景法（Scenario Method）

**定义**：基于用户使用流程，设计"主路径 + 备选路径 + 异常路径"的场景用例。适合端到端的功能流程。

**适用场景**：
- 用户故事、业务流程
- 多步骤交互
- 用于**第一章 功能测试用例**（主要）、**第五章 异常与回归用例**（部分）

**路径类型**：

| 路径类型 | 含义 | 示例（登录流程） |
|---|---|---|
| 主路径 | 用户按预期完成目标的标准路径 | 打开登录页 → 填写正确用户名密码 → 提交 → 登录成功 → 跳转首页 |
| 备选路径 | 合法但非主流的路径 | 打开登录页 → 点击"忘记密码" → 进入重置流程 |
| 异常路径 | 各种错误场景 | 密码错误 / 账号锁定 / 网络超时 / 重复提交 |

**AI 使用规则**：
- feature 任务每个核心流程必须至少覆盖：1 条主路径 + 1 条备选路径 + 3 条异常路径
- 异常路径来源：DES 的"异常处理"章节 + C-01~C-10 编码规约的异常点（空值/超时/重复提交/权限失败等）
- 生成的用例在 TC 文档中标注 `(方法论: 场景法)`

### 5. 状态迁移（State Transition）

**定义**：对有状态的对象（如订单、用户、工单），测试其在不同状态之间的合法/非法迁移。

**适用场景**：
- 订单状态（待支付 → 已支付 → 已发货 → 已签收 / 已取消 / 退款中 ...）
- 审批流（草稿 → 提交 → 审批中 → 通过 / 拒绝 / 撤回）
- 连接状态（断开 → 连接中 → 已连接 → 重连中 ...）
- 用于**第一章 业务流程测试**（数据/状态验证场景）

**设计方法**：

1. 从 DES 的"状态管理"章节提取状态机图
2. 列出所有合法迁移（每条迁移 1 条用例）
3. 列出所有**非法迁移**（每条非法迁移 1 条用例，断言"迁移应被拒绝"）
4. 列出"终态"的稳定性（终态不应被任何事件改变）

**示例**：订单状态

| 源状态 | 事件 | 目标状态 | 合法？ | TC |
|---|---|---|---|---|
| 待支付 | 支付成功 | 已支付 | ✅ | TC-F-order-001 |
| 待支付 | 超时 | 已取消 | ✅ | TC-F-order-002 |
| 已支付 | 发货 | 已发货 | ✅ | TC-F-order-003 |
| 已支付 | 取消 | 已取消 | ✅（含退款）| TC-F-order-004 |
| 已发货 | 取消 | - | ❌ 应拒绝 | TC-F-order-005 |
| 已签收 | 任何事件 | - | ❌ 终态 | TC-F-order-006 |

**AI 使用规则**：
- 有状态对象必须构造完整状态迁移表
- 每条合法迁移 + 每条非法迁移 = 一条 TC 用例
- 生成的用例在 TC 文档中标注 `(方法论: 状态迁移)`

### 5.5 图片 PRD 的 UI 用例推导（补充方法论）

当 `prd_source` 含图片（B 档：`.png` / `.jpg` / `.webp` / `.svg` / `.gif` / `.bmp` / `.tiff`）时，Skill 通过 Claude 多模态视觉分析，从图片中提取 UI 信息并推导测试用例。

**适用场景**：前端任务、UI-heavy 模块、Figma 导出的设计稿、截图评审的页面

**提取方法**：详见 `references/prd-format-handlers.md` 第二章「B 档：图片类格式」的「7 类提取信息」

**从图片到用例的映射规则**：

| 图片元素 | 对应用例章节 | 对应 channel | 对应方法论 |
|---|---|---|---|
| 页面整体布局 | 第四章 UI 测试 | `ui-visual`（exec-mode: manual） | 场景法（人工视觉对比） |
| 组件清单（输入/按钮/表格） | 第四章 UI 测试 | `ui-dom`（优先）或 `ui-visual`（降级） | 场景法 + 等价类划分 |
| 字段标签 + 占位文字 | 第二章 接口测试 + 第三章 数据/状态 | `api` + `db` | 等价类划分 + 边界值分析 |
| 交互元素的状态（禁用/加载/激活） | 第四章 UI 测试 | `ui-dom` | 状态迁移 |
| 页面状态（加载/空/错误/成功） | 第四章 UI 测试 + 第五章 异常 | `ui-dom` + `ui-visual` | 场景法（异常路径） |
| 导航路径（入口/跳转/返回） | 第一章 功能测试 | `ui-dom` | 场景法（主路径 + 备选路径） |
| 品牌视觉规格（颜色/字号/间距） | 第四章 UI 测试 | `ui-visual`（manual） | 错误推测（常见视觉走样） |

**使用规则**：
- 图片 PRD 是**补充**，不是唯一来源：若同时有文本 PRD，文本为权威，图片作视觉补充
- 图片 PRD 缺失时：第四章 UI 用例降级为基于文本描述推导（可能写不全）
- 图片冲突时：向用户询问，不强行合并
- 仅有图片无文本时：允许，但 TC 文档头部 `inferred-des: true`，并在"不测范围"章节标注"基于视觉推断"

**图片 PRD → 章节分布示例**：

假设一个登录页面，提供了 `login-normal.png` + `login-error.png` + `login-empty.png` 三张截图，Skill 从中推导：

- 第一章 功能测试：2 条（正常登录流程、忘记密码流程，均从导航路径推导）
- 第二章 接口测试：5 条（用户名/密码字段的正常/缺失/越界/权限/异常，从字段标签推导）
- 第三章 数据状态：1 条（登录失败次数锁定，从 login-error.png 的错误提示推导）
- 第四章 UI 测试：6 条（布局还原 + 按钮交互 + 3 种状态展示 + 品牌视觉规格）
- 第五章 异常回归：3 条（网络失败 / 服务器错误 / 表单重复提交）

**视觉分析的局限**：
- 无法提取后端逻辑细节（数据库约束、并发处理、事务边界）
- 无法提取性能要求（响应时间、吞吐量）
- 无法提取安全细节（加密算法、会话管理）
- 这些只能从文本 PRD 或 DES 获取

### 6. 错误推测（Error Guessing）

**定义**：基于经验和直觉，推测"容易出错的地方"并针对性设计用例。这是一种补充方法，用于捕捉前 5 种方法可能漏掉的边角用例。

**适用场景**：
- 第五章 异常与回归用例
- 补充其他方法论未覆盖的"常见坑"

**常见错误模式清单**（AI 必须逐条检查是否适用当前模块）：

| 模式 | 示例 |
|---|---|
| 空值/null 未防御 | API 字段为 null 时崩溃 |
| 并发冲突 | 两个请求同时修改同一资源 |
| 重复提交 | 用户双击按钮 / 接口重试 |
| 超时 | 上游服务超时 / 客户端等待超限 |
| 网络异常 | 请求中断 / 响应不完整 |
| 权限错位 | 用户 A 操作用户 B 的资源 |
| 数据不一致 | 缓存与数据库不同步 |
| 时间边界 | 跨天/跨月/跨年的零点操作 |
| 浮点精度 | 金额计算 0.1+0.2 != 0.3 |
| 字符编码 | Unicode / Emoji / 繁简转换 |
| SQL 注入 | 输入含 `' OR 1=1` |
| XSS | 输入含 `<script>` |
| 大小写 | URL/用户名/邮箱大小写处理 |
| 空白字符 | 前后空格 / Tab / 零宽字符 |
| 大数据量 | 列表 10000 项 / 文本 1MB |

**AI 使用规则**：
- feature 任务至少从上表中选 5 条适用的模式生成异常用例
- bug 任务重点覆盖导致 bug 的模式类型
- 生成的用例在 TC 文档中标注 `(方法论: 错误推测)`

## 二、TC-01 ~ TC-06 自审规则

这 6 条规则用于 Skill Step 5 的自审。AI 生成 TC 文档后，逐条检查，不通过当场修复。

### TC-01 验收标准覆盖

**检查内容**：REQ 中每条验收标准必须至少对应 1 条测试用例（双向追溯）。

**检查方法**：
1. 列出 REQ 的全部验收标准编号（§X.Y 格式）
2. 对每条验收标准，在 TC 文档的"覆盖率矩阵"章节查找对应行
3. 任一验收标准未被覆盖 → 不通过

**不通过示例**：
- REQ §2.1 "正常登录"、§2.2 "密码错误"、§2.3 "账号锁定"
- TC 只有 "正常登录" 和 "密码错误"，未覆盖 "账号锁定"

**修复方向**：
- 为缺失的验收标准补充对应用例
- 更新"覆盖率矩阵"，填入 TC ID
- 覆盖率统计目标：100%

### TC-02 DES 维度覆盖

**检查内容**：DES 中涉及的每个维度（数据模型/API/核心流程/状态管理/权限矩阵/UI 还原）必须有对应用例。

**检查方法**：
1. 读 DES 文档，列出涉及的维度清单（不涉及的标"N/A"）
2. 对每个涉及的维度，在 TC 文档中查找至少 1 条对应用例
3. 任一涉及的维度未被覆盖 → 不通过

**维度 → 章节映射**：

| DES 维度 | 应落在 TC 哪个章节 |
|---|---|
| 数据模型 | 第三章 数据/状态 |
| API 定义 | 第二章 接口测试 |
| 核心流程 | 第一章 功能测试 |
| 状态管理 | 第三章 数据/状态 |
| 权限矩阵 | 第二章 接口测试（权限不足用例） |
| UI 还原 | 第四章 UI 测试 |

**不通过示例**：
- DES 定义了 4 个 API，TC 文档只写了 2 个接口用例
- DES 有状态管理章节，TC 文档第三章为空

**修复方向**：
- 为每个 API 至少补充：正常请求、参数缺失、参数越界、权限不足、异常返回 5 类用例
- 为每个状态对象补充完整的状态迁移用例

### TC-03 边界值覆盖

**检查内容**：按"边界值分析"方法论，确保数值/字符串/集合/时间字段都有边界用例。

**检查方法**：
1. 从 DES 数据模型提取所有字段及其约束
2. 对每个字段，检查 TC 文档是否有边界用例
3. 任一字段的边界未被覆盖 → 不通过

**必查清单**：
- 数值字段：min / max / 0 / 负数（若允许） / 越界高 / 越界低
- 字符串字段：空 / maxLen / maxLen+1 / 特殊字符
- 集合字段：空 / 1 / maxItems / maxItems+1
- 时间字段：过去 / 未来 / 跨年 / 闰年（若涉及）

**不通过示例**：
- 年龄字段约束 1~150，TC 只测了 25，未覆盖 0、负数、>150、非数字

**修复方向**：
- 补充边界值用例
- 在覆盖率矩阵里标注 `(方法论: 边界值分析)`

### TC-04 异常路径覆盖

**检查内容**：DES 中的每个异常分支必须有对应异常用例；C-01~C-10 编码规约涉及的异常路径必须覆盖。

**检查方法**：
1. 列出 DES "异常处理" 章节中的所有异常分支
2. 从 C-01~C-10 抽取本模块适用的异常点
3. 对每个异常点，检查 TC 文档第五章是否有对应用例
4. 任一异常点未被覆盖 → 不通过

**C-01~C-10 对应的异常点清单**：

| 规约 | 异常点 | TC 建议 |
|---|---|---|
| C-01 异常路径 | try-catch 路径 | 每个 try-catch 至少 1 条用例模拟异常分支 |
| C-02 输入校验 | 非法输入 | 每个必填字段 1 条缺失用例 + 1 条越界用例 |
| C-03 幂等 | 并发 / 重复提交 | 至少 1 条并发用例 + 1 条重复提交用例 |
| C-06 超时降级 | 外部服务超时 | 至少 1 条超时用例 |
| C-07 数据边界 | null / 空数组 / 超长 / 特殊字符 | 由 TC-03 覆盖 |
| C-08 安全基线 | SQL 注入 / XSS | 至少 1 条注入用例（预期应被拦截） |
| C-10 资源释放 | 中途失败的清理 | 至少 1 条异常中断用例 |

**不通过示例**：
- DES 定义了 "库存不足" 异常，TC 未覆盖
- 模块涉及外部 API，TC 未覆盖超时情况

**修复方向**：
- 按清单补充异常用例
- 每条异常用例明确异常触发条件 + 期望行为

### TC-05 用例可执行

**检查内容**：前置条件、测试步骤、预期结果必须具体，无模糊描述。

**模糊描述黑名单**（AI 生成时禁止出现）：
- "功能正常"
- "正常运行"
- "应该没问题"
- "符合预期"
- "显示相关信息"
- "正确处理"
- "没有报错"

**检查方法**：
1. 遍历 TC 文档每条用例的"预期结果"和"测试步骤"
2. 出现黑名单词汇 → 不通过
3. 前置条件未具体到"数据/登录/环境" → 不通过

**不通过示例**：
- 预期结果写 "功能正常"
- 步骤写 "进行测试"

**修复方向**：
- 改为 "预期：页面跳转到 /order/success，地址栏参数含 orderId，toast 显示『下单成功』"
- 步骤细化到可重复执行的程度
- tc-exec 块的存在本身就强制了可执行性，这一项主要审 Markdown 描述部分

### TC-06 优先级合理

**检查内容**：P0 用例覆盖核心正向路径和主要异常；P1/P2 覆盖边界和次要场景；P0 占比不超过 50%。

**检查方法**：
1. 统计 TC 文档中 P0 / P1 / P2 的数量
2. P0 占比 = P0 数 / 总数
3. P0 占比 > 50% → 不通过（说明分级不合理，所有用例不可能都是核心）
4. P0 用例必须对应 REQ 中明确标注的核心验收标准

**优先级分配建议**：
- P0：核心主路径 + 最主要的异常（如登录成功、密码错误）
- P1：边界场景 + 次要异常（如账号锁定、session 过期）
- P2：极端边界 + 细节验证（如字段超长 10000 字符）

**不通过示例**：
- 10 条用例全标 P0
- 明显是次要场景（如国际化字符）却标 P0

**修复方向**：
- 降级明显非核心的用例到 P1/P2
- 目标分布：P0:P1:P2 ≈ 3:5:2

---

## 三、任务类型差异化检查

不同任务类型有额外的差异化检查项。

### feature 任务

- 所有 4 章都要覆盖（非前端任务第二、三章标"不涉及"）
- TC-01~TC-06 全量检查
- UI 前端改动时额外检查：每条验收标准中的 UI 部分是否有对应 `channel=ui-dom` 或 `channel=ui-visual` 用例

### bug 任务

- **必须包含 ≥1 条精确复现用例**（锁定 bug 不再出现）
  - 复现用例的步骤必须与 bug report 完全一致
  - 预期结果是"bug 修复后的正确行为"
- **必须包含回归用例**
  - 覆盖 DES "受影响 Spec 变更清单" 列出的所有模块
  - 每个受影响模块至少 1 条用例验证原有功能未受影响
- TC 文档头部 `affected-module` 字段必须填写

### refactor 任务

- **禁止依赖实现细节的用例**
  - 所有用例只能断言"输入 → 输出"的等价性
  - 禁止断言"调用了哪个内部方法"、"内部状态如何"
- **必须包含业务行为等价性断言用例**
  - 对每个对外 API / 用户可见功能，生成"重构前后行为等价"的用例
  - 用例设计从 REQ 或 DES 的"不变边界"中提取
- TC 文档头部 `affected-module` 字段必须填写

## 四、执行规则

- TC-01 ~ TC-06 **逐项检查**，不通过项当场修复
- 单项修复 **超 3 次仍未通过 → 触发停车信号**
  - 报错格式：`【TC 自审失败】项目: TC-0X | 失败原因: {原因} | 已尝试 3 次修复 | 建议: 检查 REQ/DES 完整性或人工介入`
- 一致性校验（Markdown 描述 ↔ tc-exec 块语义对齐）作为额外检查项
  - 每条用例必须逐条比对描述文字和 tc-exec 块，确保：
    - 前置条件 ↔ `**前置**` bullet
    - 测试步骤 ↔ 请求 fence block
    - 预期结果 ↔ `**断言**` bullet
  - 任一不一致 → 修复后重审
- 全部通过后 AI 标记 TC 文档头部 `test-case-status: draft`，进入 Step 6 派生单测骨架

## 五、TC 自审报告格式

TC 文档头部"自审报告"章节使用固定格式：

```markdown
## 自审报告（TC-01 ~ TC-06）

- **TC-01 验收标准覆盖**: ✅ 通过 | 覆盖率 15/15 = 100%
- **TC-02 DES 维度覆盖**: ✅ 通过 | 已覆盖维度: 数据/API/流程/状态/权限/UI
- **TC-03 边界值覆盖**: ✅ 通过 | 已修复 2 项（补充字段长度越界、特殊字符）
- **TC-04 异常路径覆盖**: ✅ 通过 | 已覆盖异常: 网络失败/超时/密码错误/账号锁定/权限不足/SQL 注入
- **TC-05 用例可执行**: ✅ 通过 | 所有用例前置/步骤/预期均具体
- **TC-06 优先级合理**: ✅ 通过 | P0:8 / P1:9 / P2:3 | P0 占比 40%

**一致性校验**: ✅ 所有用例的 Markdown 描述与 tc-exec 块语义对齐
**修复次数**: {总次数}
```

全部 ✅ 才能进入 Step 6。任一 ❌ 或 ⚠️ 都要回 Step 3/4 修复。
