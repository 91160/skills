# PRD 多格式处理指引

本文件定义 `test-case-design` Skill 对不同格式 PRD 的加载、提取、合并规则。供 SKILL.md 的 Step 1（加载输入内容）引用。

**支持范围**：A 档（文本类）+ B 档（图片类）共 13 种格式。
**明确不支持**：C 档（PDF / DOCX / RTF / ODT）、D 档（CSV / XLSX / URL）、E 档（视频 / 音频 / 加密文件）。

---

## 一、A 档：文本类格式（一等公民）

### 支持扩展名

| 格式 | 扩展名 | 读取方式 | 特点 |
|---|---|---|---|
| Markdown | `.md` / `.markdown` / `.mdx` | Read 工具直接读文本 | 章节结构清晰，表格/代码块/引用丰富 |
| 纯文本 | `.txt` | Read 工具直接读 | 平铺内容，需要 AI 识别段落和列表 |
| HTML | `.html` / `.htm` | Read 工具读取 HTML 源码 | Confluence / Notion / 飞书导出 |

### 提取规则

AI 从文本类 PRD 中提取以下信息（按优先级）：

1. **标题层级**（`#` / `##` / `<h1>` / `<h2>` 等）→ 映射到用例的"关联 REQ 章节"（如 `REQ §2.1`）
2. **验收标准清单**（通常标题含"验收"、"必须"、"应该"、"AC"、"Acceptance Criteria"）→ 每条单独识别，作为用例生成的主要来源
3. **功能描述段落** → 作为"场景法"的主路径/备选路径来源
4. **异常场景段落**（关键词：异常、错误、失败、边界、容错、降级、兜底）→ 作为"错误推测"法的输入
5. **数据模型/字段表**（Markdown 表格、HTML `<table>`）→ 提取字段名/类型/约束/必填，用于"等价类划分"和"边界值分析"
6. **API 定义**（代码块 ```http / ```json / ```yaml）→ 作为第二章接口测试的输入
7. **状态流转**（状态枚举表、状态图的文字描述）→ 作为"状态迁移"法的输入
8. **用户故事**（As a ... I want ... So that ...）→ 作为"场景法"主路径的源头

### HTML 特殊处理

HTML 文件 Read 后会得到 HTML 源码（含标签）。AI 处理时：
- 剥离 `<script>` / `<style>` 标签内容
- 保留 `<h1>` ~ `<h6>` 作为章节层级
- 保留 `<table>` / `<thead>` / `<tbody>` 作为表格结构
- 保留 `<ul>` / `<ol>` / `<li>` 作为列表
- 保留 `<code>` / `<pre>` 作为代码块
- 忽略 `<nav>` / `<header>` / `<footer>` 等导航框架元素

### A 档解析失败的降级

| 场景 | 降级 |
|---|---|
| 文件为空或字符数 <50 | 跳过该文件，记录"空文件"到合并报告 |
| 文件存在但全是图片引用（Markdown 图片语法 + 无文本） | 提示用户"该 .md 文件没有文本内容，建议提供原始图片" |
| 乱码或编码错误 | 尝试 UTF-8 → GBK 降级，仍失败则跳过并提示用户 |
| HTML 全是布局框架无正文 | 尝试提取 `<main>` / `<article>`，无则跳过 |

## 二、B 档：图片类格式（AI 视觉分析）

### 支持扩展名

| 格式 | 扩展名 | 读取方式 | 特点 |
|---|---|---|---|
| PNG | `.png` | Read 工具加载，Claude 多模态视觉识别 | 最常见，无损，适合 UI 稿 |
| JPEG | `.jpg` / `.jpeg` | 同上 | 有损压缩，照片常见 |
| WebP | `.webp` | 同上 | 现代压缩格式 |
| GIF（静态首帧） | `.gif` | 同上，仅提取第一帧 | 动图只看首帧 |
| SVG | `.svg` | Read 工具读 XML 文本 + 视觉渲染 | 矢量，可读结构 |
| BMP | `.bmp` | 同 PNG | 老格式 |
| TIFF | `.tiff` / `.tif` | 同 PNG | 印刷质量 |

### 提取的 7 类信息

AI 从图片 PRD 中"看图提取"以下信息，用于生成**第四章 UI 测试用例**：

#### 1. 页面整体布局

- 区域划分（header / main / sidebar / footer）
- 层级关系（模块 → 容器 → 组件 → 字段）
- 对齐方式（左对齐 / 居中 / 两端对齐）
- 响应式提示（如果图片标注了移动端/桌面端两种视图）

**输出**：写入 TC 文档第四章的"布局还原"用例（channel: ui-visual）

#### 2. 组件清单

- 输入框（text input、textarea、number、date）
- 按钮（primary / secondary / link / icon）
- 下拉菜单（select、multi-select、autocomplete）
- 单选 / 多选（radio、checkbox、switch）
- 表格（table、列表）
- 卡片 / 面板 / 弹窗（modal、drawer、popover）
- 导航（tabs、steps、breadcrumb、menu）

**输出**：每个组件至少生成 1 条交互用例（channel: ui-dom）

#### 3. 字段标签 + 占位文字

- 字段名称（如"用户名"、"手机号"、"密码"）
- 占位文字（placeholder，如"请输入用户名"）
- 辅助文字（help text，如"长度 4-20 位"）
- 错误文字（error message，如"用户名已存在"）
- 必填标记（红色 `*` 或 "必填"字样）

**输出**：生成第二章接口测试的**字段校验用例**（缺失/越界/格式错误，用等价类划分 + 边界值分析法）

#### 4. 交互元素的状态

- 正常态（默认可点击）
- 禁用态（灰色、不可点击）
- 加载态（loading spinner、骨架屏）
- 激活态（hover、focused、selected、pressed）
- 禁止态（error、invalid）

**输出**：每种状态至少生成 1 条 UI 状态用例（channel: ui-dom 或 ui-visual）

#### 5. 页面状态（5 态）

- 加载中（loading skeleton、spinner）
- 空数据（empty state，如"暂无订单"）
- 错误状态（error toast、错误页面）
- 成功状态（success toast、确认页面）
- 部分加载（部分成功 / 部分失败）

**输出**：每种状态至少生成 1 条状态展示用例

#### 6. 导航路径

- 入口页面 → 主流程 → 结束页
- 分支路径（从哪里跳到哪里，参数怎么传）
- 返回逻辑（有无返回按钮，返回后状态如何）

**输出**：生成第一章功能测试的**场景法**用例（主路径 + 备选路径）

#### 7. 品牌视觉规格

- 主色（primary color）
- 字号层级（H1 / H2 / body / caption）
- 间距规格（margin / padding）
- 圆角（radius）
- 阴影（shadow）
- 图标系统

**输出**：生成第四章 UI 视觉还原用例（channel: ui-visual，exec-mode: manual，人工对比截图）

### SVG 的双通道处理

SVG 文件既是文本（XML）又是图形。双通道处理：

1. **文本通道**：Read 读取 XML 源码，提取 `<text>` 元素、属性值、ID 命名 → 提取确切文字
2. **视觉通道**：同时做视觉渲染分析 → 提取布局和组件

两个通道信息合并，SVG 的提取精度通常高于位图。

### 图片提取失败的降级

| 场景 | 降级 |
|---|---|
| 图片全黑 / 全白 / 损坏 | 跳过该图，记录"视觉提取失败"到合并报告 |
| 图片分辨率过低（<200×200） | 提示用户"分辨率过低，可能无法提取细节"，继续尝试 |
| 图片内容是代码截图 | 尝试识别代码文字，提示用户"建议直接提供文本形式" |
| 图片是流程图/架构图（非 UI 稿） | 按架构图理解，提取模块/流程，不生成 UI 用例，输出到"不测范围与理由"作为参考 |
| 图片含大量手写批注 | 尝试识别批注内容，与图片主体合并 |

## 三、多文件合并规则

一个模块可以同时提供多个 PRD 文件（可以跨格式）。Skill 按以下规则合并处理：

### 优先级排序

1. **A 档文本文件** 优先（作为"权威"）
2. **B 档图片文件** 次之（作为"视觉补充"）
3. 其他格式（C/D 档）→ 不支持，走降级路径

### 合并策略

#### 多份 A 档文本
- 按文件名字典序排序（确定性）
- 同名章节：后来者覆盖前者（允许增量更新）
- 不同章节：合并到同一个 PRD 上下文

#### 多张 B 档图片
- 按文件名字典序排序
- 依次视觉分析，提取信息合并
- 遇到同一页面的多张截图（如 `login-normal.png` + `login-error.png`）→ 识别为"同一页面的不同状态"，合并到同一组 UI 用例
- 遇到不同页面的截图 → 按"页面"分组，每组独立生成用例

#### 文本 + 图片混合
- **文本作为权威**：数据模型、API 定义、业务规则以文本为准
- **图片作为视觉补充**：UI 布局、组件、状态以图片为准
- **冲突检测**：
  - 字段名冲突（文本写"用户名"、图片标注"账号"）→ 向用户询问
  - 字段约束冲突（文本说最长 20，图片标注最长 16）→ 向用户询问
  - UI 状态冲突（文本说 3 态，图片只有 2 态截图）→ 按文本生成，图片作为补充

### 合并报告模板

Step 1 合并完成后，输出：

```
【PRD 合并加载报告】
  总文件数: {N}
  按档位分布:
    - A 档（文本）: {n} 个
      · {filename.md} ({N} 字符, {M} 个章节)
      · {filename.html} ({N} 字符, {M} 个章节)
    - B 档（图片）: {n} 个
      · {filename.png} (识别到 {N} 个 UI 元素)
      · {filename.jpg} (识别到 {N} 个 UI 元素)
  合并策略: {以文本为权威 / 仅文本 / 仅图片}
  冲突: {N} 处
    · 字段"用户名"长度：文本 20 vs 图片 16 → 已向用户询问
  跳过: {N} 个
    · {filename}：{跳过原因}
  提取到的关键信息:
    - 验收标准条数: {N}
    - API 定义数: {N}
    - 数据模型字段数: {N}
    - UI 组件数: {N}
    - 页面状态数: {N}
```

## 四、不支持格式的降级路径

当用户提供的 PRD 文件不在 A/B 档时，Skill 按以下规则处理：

### C 档：PDF / DOCX / RTF / ODT

| 格式 | 降级提示 |
|---|---|
| `.pdf` | "PDF 格式不直接支持。建议 A. 复制文本内容粘贴到对话；B. 将 PDF 关键页截图为 PNG 提供；C. 先用其他工具转为 Markdown" |
| `.docx` | "DOCX 格式不直接支持。建议在 Word 中另存为 Markdown 或纯文本后重试" |
| `.rtf` / `.odt` | 同 DOCX |

### D 档：CSV / XLSX / URL

| 格式 | 降级提示 |
|---|---|
| `.csv` | "CSV 格式不直接支持。如果是验收标准矩阵或字段清单，建议转为 Markdown 表格" |
| `.xlsx` | "XLSX 格式不直接支持。建议在 Excel 中另存为 CSV 后手工整理为 Markdown" |
| `http(s)://...` | "在线链接不直接支持（避免权限/网络问题）。建议打开链接复制内容，或截图提供" |

### E 档：视频 / 音频 / 加密文件

| 格式 | 降级提示 |
|---|---|
| `.mp4` / `.mov` / `.webm` | "视频不支持。建议提供关键帧截图（A 档）+ 文字描述" |
| `.mp3` / `.wav` | "音频不支持。请提供文字稿" |
| 加密 / 登录保护的链接 | "加密或需要登录的内容不支持。请提供解密后的原始文件" |

### 降级交互示例

```
【PRD 格式不支持】
检测到文件：user-guide.pdf, requirements.xlsx

以下格式本 Skill 暂不支持：
  - user-guide.pdf (PDF)
  - requirements.xlsx (XLSX)

建议：
  1. 将 PDF 关键页截图为 PNG 后重新提供
  2. 将 XLSX 转为 Markdown 表格（手工或用 Pandoc）
  3. 如果文字内容简单，可以直接粘贴到对话中

是否：
  A. 我转换后重新调用 Skill（暂停本次执行）
  B. 跳过这些文件，用其他支持的 PRD 继续（当前还有 login-spec.md 可用）
  C. 取消本次执行
```

## 五、全部 PRD 都无法使用时

触发**停车信号**，不强行生成 TC 文档：

```
【PRD 加载失败】
所有提供的 PRD 文件都无法读取或不支持：
  - {file1}: {原因}
  - {file2}: {原因}
  ...

Skill 拒绝在无 PRD 输入的情况下猜测生成测试用例。

请：
  1. 提供 A 档（.md/.txt/.html）或 B 档（图片）格式的 PRD 文件
  2. 或直接在对话中粘贴需求文本（需 ≥200 字符，含章节结构）
  3. 如果实在没有 PRD，考虑先用 /prd-audit Skill 生成初版 PRD 后再回来
```

## 六、SDD 模式下的常见 PRD 目录

SDD 模式下，`prd_source` 参数默认指向 `.docs/prd/` 目录。该目录的常见文件组合：

| 组合 | 说明 |
|---|---|
| 仅 `.md` 文件 | 纯文本 PRD，按 A 档处理 |
| 仅 `.png` / `.jpg` 文件 | UI 设计稿，按 B 档处理，适合前端任务 |
| 混合 `.md` + 图片 | 文本说明 + UI 参考图，按"文本权威 + 图片补充"合并 |
| `.pdf` 文件 | 不支持，走 C 档降级路径提示用户 |

SDD 模式下，Skill 通过 `project-profile.md` 的「PRD 内容索引」定位当前模块相关的 PRD 文件（不读无关模块的 PRD）。

## 七、处理超大文件

| 大小 | 处理 |
|---|---|
| 文本文件 <100KB | 正常读取 |
| 文本文件 100KB~1MB | 读取前 500KB + 尾 100KB，中间部分抽样，输出截断提示 |
| 文本文件 >1MB | 提示用户"PRD 过大，建议按模块拆分" |
| 图片 <2MB | 正常读取 |
| 图片 2MB~10MB | 正常读取但告知可能影响性能 |
| 图片 >10MB | 提示用户"图片过大，建议压缩或裁切后重试" |

## 八、格式处理清单（Step 1 执行时逐文件检查）

Skill 对每个 `prd_source` 文件执行：

1. [ ] 检查扩展名是否在 A/B 档支持清单内
2. [ ] 支持 → 按对应规则读取和提取
3. [ ] 不支持 → 走降级路径提示用户
4. [ ] 读取成功 → 提取关键信息（按 A 档 8 项或 B 档 7 类）
5. [ ] 读取失败 → 记录到合并报告的"跳过"清单
6. [ ] 合并到全局 PRD 上下文
7. [ ] 全部文件处理完毕 → 输出合并报告
8. [ ] 全部失败 → 触发停车信号
