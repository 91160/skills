# 质量标准体系

各阶段产出物的审计标准统一定义如下，具体执行时机见对应步骤引用。

---

## PRD 审计标准

| 维度 | 检查内容 | 不通过示例 |
|------|---------|-----------|
| 完整性 | 正常 + 异常流程是否完整 | 只描述了"用户下单"，未说明库存不足、支付失败、超时取消的处理 |
| 一致性 | 与现有功能/数据模型是否冲突 | 新增"会员等级"字段，但用户表已有"vip_level"字段，定义冲突 |
| 可行性 | 现有架构是否支撑 | 要求实时推送消息，但项目无 WebSocket 基础设施 |
| 边界 | 做什么/不做什么是否明确 | "支持多种支付方式"——具体哪几种？微信/支付宝/银行卡？ |
| 优先级 | P0/P1/P2 是否明确 | 10 个需求全标 P0，无法区分优先级 |
| 安全合规 | 权限/数据安全/隐私是否覆盖 | 涉及手机号展示但未说明脱敏规则 |

P0 问题必须解决后才继续。

## REQ 审计标准

| 维度 | 检查内容 | 不通过示例 | 修复方向 |
|------|---------|-----------|---------|
| 验收标准具体性 | 每条验收标准是否可验证，无模糊词 | "页面加载正常" → 无法验证 | 改为"页面在 3 秒内完成首屏渲染，无控制台报错" |
| 边界明确性 | 做什么/不做什么是否清晰 | "支持搜索功能" → 搜索什么？模糊搜索还是精确？支持哪些字段？ | 明确"按商品名称模糊搜索，支持 name/sku 两个字段" |
| 与 PRD 一致 | 是否忠实于 PRD 对应部分 | PRD 要求"仅管理员可删除"，REQ 验收标准未提及权限 | 补充"非管理员角色点击删除按钮，提示权限不足" |
| 异常场景 | 异常/边界场景是否至少列出 | 只写了正常提交流程，未列出网络异常、重复提交、字段超长 | 补充异常场景清单，每个场景有预期行为 |

## DES 审计标准

| 维度 | 检查内容 | 不通过示例 | 修复方向 |
|------|---------|-----------|---------|
| 维度覆盖 | feature 类型的 5 维度（数据模型/API/核心流程/状态管理/权限）是否涉及项均已覆盖 | DES 只写了 API 定义和前端页面，未定义数据模型字段类型和约束 | 补充数据模型：字段名/类型/长度/约束/默认值/索引 |
| 与 REQ 一致 | 设计方案是否覆盖 REQ 全部验收标准 | REQ 有"超长文本截断"验收标准，DES 未定义截断规则（多少字？省略号？tooltip？） | DES 中增加"文本超过 50 字时截断，显示省略号，hover 展示 tooltip" |
| 可编码性 | 读完 DES 即可编码，无需猜测 | "使用合适的组件展示列表" → 哪个组件？van-list？el-table？无限滚动？ | 明确"使用 van-list 组件，下拉加载，每页 20 条，加载完毕显示'没有更多了'" |
| 异常穷举 | 异常路径是否完整定义 | API 错误码只列了 200 和 500，未定义 401/403/404/422 | 列出所有可能的错误码及其触发条件和前端处理方式 |
| 架构一致 | 方案与 context 文件（frontend-context.md / backend-context.md）一致；复用能力清单正确匹配 context 文件中已有的内部公共能力 | 自定义了一个 request 工具函数，但 frontend-context.md 中已有 `@/utils/request.ts` | 使用 context 文件中已有的 HTTP 封装，删除重复实现 |
| PRD 一致 | 前端 DES 的布局/组件/交互/字段是否覆盖了 `.docs/prd/` 中的全部内容（含视觉内容），偏离处是否标注原因 | PRD 图片中有 4 个表单字段，DES 只定义了 3 个，遗漏了"备注"字段 | 补充遗漏字段，或标注"备注字段经确认本期不做" |

## 代码审计标准（S-01 ~ S-07）

| # | 审计项 | 检查内容 | 不通过示例 | 修复方向 |
|---|--------|---------|-----------|---------|
| S-01 | 空值防御 | null/undefined/空数组是否有防御 | `data.list.map(...)` 未判断 `data.list` 是否为 null | 改为 `(data.list \|\| []).map(...)` 或 `data.list?.map(...) ?? []` |
| S-02 | 前端输入信任 | 后端是否独立校验，不信任前端 | 后端接口直接取前端传的 `userId` 查数据，未校验是否为当前登录用户 | 后端从 token 中提取 userId，忽略前端传值 |
| S-03 | Spec 一致性 | 实现是否与 REQ + DES 一致 | DES 定义接口返回 `{ code, msg, data }`，实际返回 `{ success, message, result }` | 统一为 DES 定义的格式 |
| S-04 | 性能回归 | 是否引入慢查询/大循环/内存问题 | 在 for 循环中逐条执行 SQL 查询（N+1 问题） | 改为批量查询：`WHERE id IN (...)` |
| S-05 | 权限校验 | 接口权限是否正确配置 | 新增的管理端接口未添加 `@RequiresPermission` 注解 | 按 DES 权限矩阵添加对应注解 |
| S-06 | 能力复用 | 是否按 DES「复用能力」清单使用了 context 文件中已有的内部公共能力 | 自行 `import axios from 'axios'` 发请求，但 frontend-context.md 中已有 `@/utils/request.ts` 封装 | 改为 `import { instance } from '@/utils/request'` |
| S-07 | UI 还原度 | 前端代码的页面结构/组件/交互/字段是否与 PRD 内容 + DES UI 方案一致（U-01~U-06） | PRD 图片中按钮在表单下方居中，代码中按钮放在了右上角 | 调整布局与 PRD 一致，或在 DES 中标注偏离原因后按 DES 执行 |

**执行规则**：
- 逐项检查，不通过项**当场修复**
- 修复后重新检查该项，直到通过
- 单项修复超 3 次仍未通过 → 触发停车信号
- 全部通过后输出审计报告，追加到 `.outdocs/audit-report.md`

**审计报告格式**：

```
【代码审计报告】
结论：PASS
检查项：{N}/7 通过 | 修复：{N} 项

- S-01 空值防御：✅
- S-02 前端输入信任：✅（已修复：后端改为从 token 提取 userId）
- S-03 Spec 一致性：✅
- S-04 性能回归：✅
- S-05 权限校验：✅
- S-06 能力复用：✅
- S-07 UI 还原度：✅（前端改动时检查）
```

## 开发自测标准（T-01 ~ T-06）

**前端验证：**

| # | 检查项 | 检查内容 | 自测方法 |
|---|--------|---------|---------|
| T-01 | 交互逻辑 | 改动点涉及的每个页面中，相关按钮的交互逻辑是否正常 | 逐个点击改动涉及的按钮/表单/弹窗，验证响应符合 DES 定义 |
| T-02 | 页面跳转 | 改动涉及的页面跳转路径是否正确 | 从入口页开始，走完所有跳转路径，确认目标页正确且参数传递无误 |
| T-03 | 页面渲染 | 改动涉及的页面是否正常渲染，无白屏/报错 | 打开所有涉及页面，检查控制台无 error，页面元素完整渲染 |

**后端验证：**

| # | 检查项 | 检查内容 | 自测方法 |
|---|--------|---------|---------|
| T-04 | 接口可用性 | mock 数据验证改动涉及的所有接口，保证不报错 | 用 mock 数据调用每个接口，确认返回 200 且结构符合 DES |
| T-05 | 边界覆盖 | 覆盖空值、超长、非法输入等边界场景 | 传入 null、空字符串、超长文本（>255字）、特殊字符（`<script>`）、负数、0，验证返回合理错误码 |
| T-06 | 返回值正确性 | 返回值结构和状态码符合 DES 定义 | 对比 DES 中 API 定义的 response 结构，逐字段验证类型和值 |

**执行规则**：
- 优先读取 `.project/specs/rules/unit-test-base.md`（团队自定义自测规则）
- 文件不存在 → 使用上述 T-01~T-06
- 不通过项回 Step 4 修复代码，修复后重新执行 Step 5（代码审计）+ Step 6（自测）
- **自愈循环上限**：Step 4→5→6 累计不超过 5 次，超过 → 触发停车信号

**自测报告格式**：

```
【开发自测报告】
结论：PASS
检查项：{N}/6 通过 | 修复：{N} 项 | 回归次数：{N}

[前端验证]
- T-01 交互逻辑：✅ 验证页面：{页面列表}
- T-02 页面跳转：✅ 验证路径：{路径列表}
- T-03 页面渲染：✅

[后端验证]
- T-04 接口可用性：✅ 验证接口：{接口列表}
- T-05 边界覆盖：✅ 覆盖场景：{场景列表}
- T-06 返回值正确性：✅
```
